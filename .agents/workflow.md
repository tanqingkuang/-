# 通用代码工作流

## 1. 执行约束

- 待办按可独立验证的工作单元拆分，不按零碎改动拆分；同类低风险小改动合并到相邻待办。
- 每步完成后先自检：`git diff`、受影响模块、关键路径回归点。
- 按风险执行测试：
  - **免测改动**：仅文档、纯注释、说明文字，不改变代码语义、配置和构建输入时，只做 `git diff` 自检。
  - **低风险**：最小必要检查（`python -m compileall -q src` 等基础检查）。
  - **中风险**：相关模块 LLT 子集（`pytest tests/llt -q -k <关键字>`）。
  - **高风险**（改动仿真/算法/runner 或跨模块接口）：完整收口，见 `.agents/test.md`。
- 若当前步骤失败，先修复再继续，不带病推进。

## 2. 检视问题处理（TDD）

处理代码检视、review 文档或缺陷反馈时，默认采用 TDD 手段：

1. 先为每个可执行的检视问题补充能失败的测试，证明问题存在或锁定预期行为。
2. 再修改实现，使新增测试和既有测试通过。
3. 若某条意见不适合用测试覆盖，必须在对应 review 文档或回复中说明原因。
4. 修复后在 review 文档中逐条回复"已修改"或"不修改原因"，并列出对应测试用例。

## 3. 最终收口

- 只要发生行为相关代码改动，最终收口不得省略；收口内容以 `.agents/test.md` 为准。
- 纯文档或纯注释改动可免收口，但必须在最终说明中写明"未改变代码语义，仅做 diff 自检"。
- 最终交付回复必须逐项列出各收口检查的 **通过 / 跳过（含原因）/ 失败（含处置）** 状态。
