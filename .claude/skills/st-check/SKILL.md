# 自动化 ST 验收

调用 `scripts/run_st.py` 对全部 ST 场景执行三层检查，并按**分层语义**处置结果。设计与用例清单见 `docs/11-自动化ST设计与用例清单.md`，阈值集中在 `tests/st/support/thresholds.py`。

## 触发边界

- **必须执行**：AI 修改了 `src/` 下仿真、算法或 runner 代码后的最终收口（见 `.agents/test.md`）；或用户明确要求跑 ST / 验证回归 / 刷基线。
- **可跳过**：本次改动只涉及 GUI、文档等与仿真结果无关的内容时可跳过，但最终回复必须说明"未运行 ST，因为未修改仿真相关代码"。
- **禁止**：为了让 ST 通过而放宽 `thresholds.py` 阈值；阈值整定只能由用户决定。

## 三层语义（处置的依据）

| 层 | 用例 | 含义 | 失败时的处置 |
|----|------|------|--------------|
| T1 不变量 | UT-01~07 | 物理限幅/安全/任务完成，**任何算法改动都必须成立** | 视为真实缺陷，必须修复，禁止刷基线绕过 |
| T2 指标回归 | UT-08~09 | 完成时间/安全裕度恶化超容差 | 确认是否可接受的性能取舍；不可接受则修复 |
| T3 轨迹基线 | UT-10 | 与基线轨迹逐点比对 | 若本次改动**有意**改变了算法行为→属预期失败，走刷基线流程；否则是意外副作用，必须排查 |

## 执行步骤

1. 在项目根目录运行：

   ```bash
   python scripts/run_st.py
   ```

2. 看**退出码**与输出判定：
   - 输出 `ST OK`、退出码 `0` → 验收通过。
   - 退出码 `1` → 逐条读失败行 `[场景][UT-xx] … @t=…s node=… field=…`，按上表分层处置。
   - 输出中的 `HINT` 行表示指标变好，不影响通过，可顺带提示用户是否刷新基线。

3. 刷新基线（**仅当** T3 失败且确认改动属预期、且 T1 全部通过时）：

   ```bash
   python scripts/run_st.py --update-baseline   # 输出 BASELINE UPDATED
   python scripts/run_st.py                     # 复跑确认 ST OK
   ```

   刷新后 `tests/st/baselines/*.json` 的 diff 要随代码一起提交，并在交付说明中写明刷新原因。

## 失败处理

- T1 失败：立即停止交付流程，先修复实现，修复后复跑；任何情况下不得用刷基线或改阈值绕过。
- T2 失败：先向用户说明恶化幅度与原因，确认可接受才放行，否则修复。
- T3 失败：先判断是否预期行为变化，拿不准时向用户确认后再决定刷基线还是排查副作用。
- 脚本自身报错（非用例失败）：报告关键错误行，检查场景 / 基线文件完整性，不要盲目重试。

## 输出契约

- 过程输出：成功时不要复述场景明细，只保留结论；失败时只引用失败行和分层归类，不贴全量日志。
- 最终汇报固定包含：
  - 结论：`st=PASS` / `st=FAIL` / `st=BASELINE_UPDATED`（或跳过及原因）。
  - 失败时按 T1 / T2 / T3 分层列出条目和你的处置。
  - 刷基线时说明原因和涉及的基线文件。

## 说明

- 单场景调试：`python scripts/run_st.py --scenario st_line`（场景名 = `tests/st/scenarios/` 下文件名去 `.json`）。
- 全套约 12s，无需后台运行。
- 新增 ST 场景的步骤见设计文档 6.5 节。
