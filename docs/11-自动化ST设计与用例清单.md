# 自动化 ST 设计与用例清单

## 1. 设计原则

- **非侵入**：不改动 runner/数据记录逻辑。ST 只消费仿真已有的落盘产物
  （`logs/run-seed-<seed>-*/snapshots_seed_<seed>.jsonl`、`events.jsonl`、`config.json`），压缩提取由独立脚本完成。
- **契约一致**：坐标、转弯、滚转和误差符号遵循 [`docs/0-坐标系约定.md`](0-坐标系约定.md)；ST 直接校验日志原值，不套用 UI 显示层的符号翻转。
- **分层防误报**：
  - T1 不变量——无论算法怎么改进都必须成立（物理限幅、安全、任务完成），改好算法零误报；
  - T2 指标——只报"恶化方向超容差"，防算法悄悄变差；
  - T3 紧凑基线——固定 seed 逐点比对，定位"非算法改动的意外副作用"，改算法时预期失败并一键刷新。
- **报警即定位**：失败输出直接给出 场景/时刻/机号/字段/差值，节省人工与 AI 排查成本。

## 2. 组件与目录

| 组件 | 路径 | 职责 |
|---|---|---|
| 轨迹压缩提取器 | `tests/st/support/trajectory_extract.py` | seed 快照 JSONL → 规范化紧凑轨迹（1s 降采样、x/y/z/航向/速度 5 字段、厘米级舍入、固定字段序、剔除 run-id/路径/时间戳等易变字段） |
| 不变量断言库 | `tests/st/support/invariants.py` | T1 各项检查，输入全量快照序列 + config |
| 指标提取与比对 | `tests/st/support/metrics.py` | T2 标量指标计算、与基线按"恶化方向 + 容差"比对 |
| 基线比对器 | `tests/st/support/baseline_diff.py` | T3 紧凑轨迹逐点比对，输出差异 top-N 定位 |
| ST 场景配置 | `tests/st/scenarios/*.json` | 现有 configs 的缩时副本（≤200s，固定 seed），仿真主体配置不动 |
| 基线数据 | `tests/st/baselines/<场景>.json` | T3 紧凑轨迹 + T2 指标基线，入 git，可读可 diff |
| 一键入口 | `scripts/run_st.py` | 顺序跑全部场景 → 三层检查 → 全过则输出 `ST OK` 并 exit 0；失败则输出定位报告并 exit 1。`--update-baseline` 刷新 T2/T3 基线，`--scenario <名>` 单场景运行 |
| pytest 接入 | `tests/st/test_st_scenarios.py` | 同一套检查以 pytest 参数化呈现，供 CI/习惯 pytest 的场合 |

运行方式：每个场景只跑一次 `SimController.run_until_complete(config, seed=固定值)`，
产物在三层检查间共享；全套 ST 目标控制在 2 分钟内（场景时长实现时实测后微调）。

## 3. ST 场景（首期 4 个，正交覆盖）

| 场景 | 来源 config | 覆盖能力 |
|---|---|---|
| `st_line` | `configs/element/line.json` 系 | 直线航线跟踪基础 |
| `st_formation` | `configs/aifly.json`（或 diamond） | 编队保持/队形变换 |
| `st_avoidance` | `configs/single_avoidance_80km.json` | 避障（含贴障弧） |
| `st_multi10` | `configs/quadrilateral_10_aircraft_a05_leader.json` | 10 机多机规模 + 非首机长机 |

## 4. 用例清单

### T1 不变量（UT-01～07，均参数化到 4 个场景）

| 编号 | 名称 | 检查内容 |
|---|---|---|
| UT-01 | 运行完成性 | `run_until_complete` 返回 OK；`events.jsonl` 无 ERROR（WARN 白名单制）；仿真推进到配置时长；快照帧数 = 时长/步长（±1） |
| UT-02 | 数值健康 | 全字段无 NaN/Inf；相邻帧位移 ≤ v_max×dt×1.5（无瞬移）；高度/速度无单帧跳变 |
| UT-03 | 动力学限幅 | 每帧 speed ∈ [v_min, v_max]、`nx ∈ [nx_min,nx_max]`、`n_normal ∈ [n_normal_min,n_normal_max]`、`phi ∈ [phi_min,phi_max]`（右倾为正）、\|air_psi_dot\| ≤ 空速角速率限幅；`nz` 是有符号右向分量，不直接套合量门限，地面航迹率也不套空速滚转包线；限幅值从 config 读取，不硬编码 |
| UT-04 | 机间防撞 | 全程任意两机 3D 间距 > 防撞阈值 |
| UT-05 | 避障有效性 | 全程与每个障碍（含膨胀半径）的水平距离 ≥ 0（仅避障场景启用） |
| UT-06 | 任务完成 | 仿真末端与航线终点的水平误差 < 阈值；末端速度方向与末航段方向夹角 < 阈值 |
| UT-07 | 编队收敛与保持 | 进入稳态段（后 50% 时间）后队形误差均值与末端值 < 阈值（仅编队场景启用） |

### T2 指标回归（UT-08～09，参数化到 4 个场景，基线入 git）

| 编号 | 名称 | 检查内容 |
|---|---|---|
| UT-08 | 完成时间不劣化 | 到达终点用时相对基线恶化不超过 +10%；变好则提示可刷新基线（不失败） |
| UT-09 | 安全裕度不缩水 | 全程最小机间距（避障场景另含最小障碍裕度）相对基线缩水不超过 −10% |

### T3 紧凑基线比对（UT-10，参数化到 4 个场景）

| 编号 | 名称 | 检查内容 |
|---|---|---|
| UT-10 | 轨迹基线一致 | 紧凑轨迹（1s 采样 × 每机 × x/y/z/航向/速度，厘米舍入）与基线逐点一致；失败输出差异 top-10（时刻/机号/字段/基线值/当前值/差值），并提示"算法改动属预期，用 `--update-baseline` 刷新" |

### ST 工具自身的 LLT（UT-11～13，`tests/llt/test_st_support.py`，不跑仿真）

| 编号 | 名称 | 检查内容 |
|---|---|---|
| UT-11 | 提取器规范化 | 降采样/舍入/字段序正确；易变字段被剔除；同一 jsonl 两次提取结果逐字节一致（确定性） |
| UT-12 | 比对器定位 | 人工注入单点差异 → 报告准确定位到该时刻/机号/字段；舍入容差内的差异不报 |
| UT-13 | 入口脚本契约 | 全过路径输出 `ST OK` 且 exit 0；失败路径 exit 1 且报告含场景与定位信息；`--update-baseline` 后重跑即通过（用微型 jsonl 夹具，不跑真仿真） |

## 5. 数量汇总与边界

- 用例类型 13 个；T1/T3 参数化后约 **7×4 + 2×4 + 1×4 + 3 ≈ 43 项**（避障/编队专属项按场景裁剪）。
- 首期不做：GUI/3D 渲染类 ST（LLT 已覆盖 payload 契约）、性能基准、跨平台一致性。
- 阈值原则：T1 阈值从 config 派生或取物理常识值，T2 容差统一 10% 起步，均集中在一个常量表便于整定。

## 6. 使用方式

### 6.1 一键运行

默认运行全部 ST 场景，并依次执行 T1 不变量、T2 指标回归、T3 紧凑轨迹基线比对：

```bash
python scripts/run_st.py
```

全部通过时只输出：

```text
ST OK
```

若失败，每条失败一行，包含场景、用例编号、时间、节点、字段、实际值和限值。例如：

```text
[st_line][UT-10] 轨迹基线不一致 @t=1.000s node=S01 field=x_m baseline=45.5 current=45.0 diff=-0.500
[st_line][UT-10] 若本次改动涉及算法属预期失败，运行 `python scripts/run_st.py --update-baseline` 刷新
ST FAILED: 2 项
```

### 6.2 单场景运行

只运行某个场景时使用：

```bash
python scripts/run_st.py --scenario st_line
```

场景名对应 `tests/st/scenarios/<场景>.json` 的文件名，不带 `.json` 后缀。

### 6.3 刷新基线

当算法改动导致轨迹或指标按预期变化时，先确认 T1 不变量仍通过，再刷新 T2/T3 基线：

```bash
python scripts/run_st.py --update-baseline
```

也可以只刷新单个场景：

```bash
python scripts/run_st.py --scenario st_line --update-baseline
```

刷新后会写入 `tests/st/baselines/<场景>.json`，文件只包含 `metrics` 和 `trajectory` 两段。

### 6.4 阈值整定入口

所有 ST 阈值集中在：

```text
tests/st/support/thresholds.py
```

包括防撞距离、终点误差、末端航向夹角、队形误差、瞬移裕度、T2 指标容差、T3 采样周期/舍入位数/字段清单。需要放宽或收紧 ST 判据时只改这个文件。

### 6.5 新增场景步骤

1. 在 `tests/st/scenarios/` 新增 `st_<name>.json`。
2. 若场景依赖航线、障碍或队形元素，把元素复制到 `tests/st/scenarios/element/`，并使用相对路径引用，避免依赖 `configs/` 产品配置。
3. 固定 `seed`，并把 `duration_s` 调到末帧接近任务终点；当前控制器到达终点后会继续按航线方向飞，ST 的 UT-06 检查末帧，因此不能让场景在终点后空跑太久。
4. 运行 `python scripts/run_st.py --scenario st_<name> --update-baseline` 生成基线。
5. 运行 `python scripts/run_st.py --scenario st_<name>`，确认输出 `ST OK`。

### 6.6 当前实现偏差说明

- 当前控制器落盘的 `snapshots_seed_<seed>.jsonl` 是 10Hz 固定采样（`0.1s`），不是按 `step_s` 每积分步写一行。因此 UT-01 的帧数检查按日志采样周期校验；瞬移检查也按相邻日志帧的实际 `dt` 计算。
- `st_avoidance` 使用 support runner 在无头运行前调用现有避障规划器并通过公开 `apply_avoidance_route()` 采用规划航线；这是为了保持 ST 非侵入，同时覆盖“障碍触发后采用绕障航线”的行为。
- 当前控制器没有配置驱动的自动队形切换事件；`st_formation` 保留 5 个队形文件用于覆盖多队形配置解析，自动切换不在本次非侵入 ST 中实现。
