# 编队算法 LLD 综述

> 本册统管三个 LLD（实体组 / 算法库 / 流程库）的**接口契约**与**统一数据契约**，并汇总跨文档 TODO。
> 架构见 `0-HLD.md`；各模块实现见对应 LLD；方案组合见 `5-用例-领航跟随保持.md`。

# 一、对外 API（编队算法 ↔ 仿真控制）

> 把编队算法当黑盒：仿真控制唯一需要知道的就是这套接口与流动的数据。

## 1.1 统一实体契约（对仿真控制）

仿真控制对所有实体用同一套契约，不区分协调 / 节点（见 `../1-仿真控制HLD.md` §8.4）：

```python
class FormationAlgorithm:
    def init(entity_id: str, config: dict[str, object]) -> None: ...  # config.entity_type ∈ {flight, coordination}
    def step(context: FormationAlgorithmContext) -> FormationAlgorithmOutput: ...
    def reset() -> None: ...
    def close() -> None: ...
```

- 实体总数 = **飞行实体 ×N** + **非飞行协调实体 ×0/1**；由 `config.entity_type` 区分（`flight` 绑 `aircraft_id`、产 control；`coordination` 不绑、不产 control）。

## 1.2 数据契约（编队算法 ↔ 仿真控制）

代码实现可用 dataclass / TypedDict 承载。锚定领航-跟随、3 自由度。

### 1.2.1 NavState — 注入算法的本机导航视图

由仿真控制从 `../2-模型迭代HLD.md` 的 `AircraftState` 投影得到，只保留真实机载导航可得量；**不含**模型内回路状态 `ax/ay/az(+rate)` 与质点输入 `nx/nz/phi`。噪声 / 漂移叠加在此投影层。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `node_id` | `str` | 本机 ID，沿用 `AircraftState.node_id`；对 `flight` 实体即其 `aircraft_id`（亦即 `init` 的 `entity_id`） |
| `x_m / y_m / altitude_m` | `float` | 东 / 北 / 天 位置（E/N/U，沿用 `AircraftState` 工程字段名） |
| `speed_mps` | `float` | 速度 V |
| `theta_rad` | `float` | 航迹倾角 |
| `psi_rad` | `float` | 航向（`0`=东，`π/2`=北，与 2- 一致） |

> 实体内部把 `NavState` 称作 **运动状态**（黑板叶类型），并区分两种引用：本机自己的叫 **本体运动状态**（own）；广播来的长机的叫 **主机运动状态**（host，对长机自身 host==own）。见 §2.4。

### 1.2.2 FormationAlgorithmContext — `step` 入参

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `time_s` | `float` | 当前仿真时间 |
| `dt_s` | `float` | 本拍步长（本轮所有实体同 20 Hz，见 `0-HLD.md` §4 调度；真·多速率留半物理） |
| `self_state` | `NavState` | 本机导航视图 |
| `inbox` | `list[MessageEnvelope]` | 上一轮派发到本实体的消息 |
| `mission_command` | `MissionCommand \| None` | **仅注入承载协调能力的实体**（本轮长机）；其任务编排产 `Mode` 并经广播下发。飞行僚机**不消费**它，僚机 `Mode` 由 `收消息` 从广播解析（见 §2.4）。本轮恒"保持"，可为 `None` |

> 静态参数（增益、队形几何表 `队形信息`、`飞机ID`、`entity_type`、网络拓扑等）走 `init`，不进 context。

### 1.2.3 FormationAlgorithmOutput — `step` 返回

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `control` | `AccelerationCommand \| None` | ENU 三轴期望加速度，对齐 2- 的 `AccelerationCommand{ax/ay/az_cmd_mps2}`；`coordination` 时 `None` |
| `outbox` | `list[MessageEnvelope]` | 本实体要发的消息 |
| `status` | `AlgorithmStatus` | 摘要：当前模态、关键误差，供日志与控制回报 |

### 1.2.4 消息 envelope

通信功能只认通用 `MessageEnvelope{topic, source, target, timestamp, payload}`，不解析 payload。各方法的 payload 由配对的算法插件自行约定并拥有（领航-跟随见 `5-用例-领航跟随保持.md`）。

# 二、对内接口（实体内 单元 ↔ 单元：黑板 + 类型化视图）

> 翻进黑盒内部：实体内数据走 **黑板 + 类型化指针视图（4 层）**，单元的"接口" = 它在黑板上读写哪些槽。
> 架构动机见 `0-HLD.md` §5（数据视图）与原则 8/10。

## 2.0 数据流总架构：4 层黑板

实体内**不再"单元间显式传值"**：所有过程数据放进一个 per-entity 的**黑板**，每个单元通过自己的**类型化指针视图**读写黑板。分 4 层：

| 层 | 内容 | 例 |
| --- | --- | --- |
| **L4 叶类型** | 可拼装的基础 `struct` / `enum` | `运动状态`(=NavState)、`航段`、`目标指令`、`三轴误差`、`编队模式`(Mode)、`队形指令` |
| **L3 黑板（`Context`）** | 把过程中所有中间量装成**一个大 struct**（值语义），单一存储；**per-entity 一份** | `Context{ 本体运动状态, 主机运动状态, 航段, 目标指令, 待飞距, 三轴误差, 编队模式, 队形指令, … }` |
| **L2 单元（算法 / 流程）** | 各单元把 `u` / `y` 定义成**一组指向黑板槽的指针**；`step(self)` 经这些指针读写黑板 | `位置解算·航线插值In{ 航段*, 本体运动状态* }` / `Out{ 目标指令*, 待飞距* }` |
| **L1 实体（对象组）** | ① 实例化黑板 ② 实例化各单元 ③ 把各单元 `u`/`y` 指针**绑到黑板槽**（init 一次）；`step()` 按**固定拓扑序**调用各单元 | 见 §2.3、`2-实体组LLD.md` |

要点：

- **没人再"搬运 / 拼装"数据**：上游写进黑板槽、下游从同一槽读，数据自然流动；`step()` 不组装 `u`（绑定在 `init` 一次完成）。
- **per-instance、非全局**：黑板与各单元视图都是**实体对象的成员**，一个实体一份；N 个僚机各持各的黑板（可重入靠实例化，原则 6）。
- **接口仍显式**：单元的 In/Out 指针结构体就是它的依赖声明，从签名能看出它读谁、写谁（**不退化成不透明的大 ctx**）。
- **跨实体边界量**：`发消息(Outbound)` 的产物 `outbox` 是黑板的**边界槽**（`Outbound.step(self)` 写它、`_emit` 取它），与其他槽一视同仁；`收消息(Inbound)` 的 `inbox` **不入黑板**——它是"用完即弃"的瞬时输入，故 `收消息` 取**唯一例外签名 `step(self, inbox)`**，其余单元一律 `step(self)`。两者内容类型都是 `list[MessageEnvelope]`。
- **C 映射**：`Context` ≡ 结构体；单元 In/Out ≡ 内含指针的结构体；`step(self)` ≡ `step(self*)` 经指针读写；绑定 ≡ 把指针指向 `Context` 成员。Python：黑板装可变叶对象，单元持有指向叶对象的引用（共享粒度 = 叶 struct，非单个标量）。

## 2.1 通用单元模板：`init / reset / step(self) / read_state`

每个单元是一个类，遵循统一模板：

```python
class Unit:                              # Protocol / 抽象基类
    def init(self, params, bind) -> None: ...  # 拉起：注入静态参数；把 u/y 视图指针绑到黑板槽
    def reset(self) -> None: ...               # 回到初始（清内部状态）
    def step(self) -> None: ...                # 推进一拍：经已绑定的 u/y 视图读写黑板
    def read_state(self) -> dict: ...          # 仅供落盘 / 快照：读内部状态 x（≠ 黑板）
    # 可选：close()
```

- **`step(self)` 无入参、无返回**：输入输出都经**已绑定的黑板视图**完成（与早先 `step(u)->y` 的区别：u/y 从"传值"变成"指向黑板的视图"）。**唯一例外**：`收消息(Inbound)` 取 `step(self, inbox)`——`inbox` 是用完即弃的瞬时输入、不进黑板（产物写黑板槽）。
- `init` / `reset` / `read_state` 是通用接口；**设计的真正内容在每个单元的 In/Out 视图声明**——即 §2.4 逐单元定义。

## 2.2 抽象基类与策略（同族可换：共性父类、差异子类）

**每个单元都是一个策略族**：族 = 一个抽象基类（统一 `step(self)` + 一组**输出槽契约**），情景差异靠**族内换实现**、非散落 `if/else`。选哪个实现由**实体挂接时**决定（原则 10）。

u/y 视图按"**共性提父类、差异下沉子类**"组织：

- **输入（u）多为 impl 私有 → 下沉子类**：每个实现只声明它真读的黑板槽，**不为同族其他实现"拉齐"补字段**（`航线插值In = {航段*, 本体运动状态*}`；`槽位几何In = {队形指令*, 主机运动状态*}`，互不背对方的字段）。
- **输出（y）多为共性 → 留父类**：尤其**下游必读的输出槽**留父类，类型层面焊死"换实现也有人写"（swap 安全）；真有差异再下沉。
- **族由"统一 step + 输出槽契约"统一（≈ 同 `y`）**，输入各取所需。
- **swap 仍成立**：impl 私有的 In/Out 只在 **init 绑定那一下**用到（那本就是选定 impl 的地方）；`step` 派发统一 `unit.step()`，实体不拼装、不需知道具体类型。C 里基类 `void (*step)(Unit* self)`，各 impl 把 `self->in/out` cast 成自己的视图类型。

| 抽象基类（族） | 库 | 同族可换实现（示例） |
| --- | --- | --- |
| `Inbound`（收-收消息） | 流程 | 领航-跟随解析 / 点对点 / … |
| `Outbound`（发-发消息） | 流程 | 领航-跟随广播 / 点对点 / 不发 / … |
| `Orchestrate`（任务编排） | 流程 | 常量"保持" / 真实模态状态机 / … |
| `TrajectoryPlan`（轨迹规划） | 流程 | 航线推进（长机）/ 空规划（僚机预埋）——**mode-aware** |
| `PositionSolve`（位置解算） | 算法 | 航线插值（长机）/ 槽位几何（僚机）/（未来）Dubins |
| `DeviationCalc`（误差解算） | 算法 | 不同偏差口径 |
| `Tracking`（编队跟踪） | 算法 | 不同通道组合（垂向 TECS / 单通道 PID…），**组合控制算法而成** |
| `ControlLaw`（控制算法） | 算法 | PID / L1 / ADRC（原子控制律，被 `Tracking` 组合调用） |

> **算法 / 流程分界 = 是否碰 `Mode`**（原则 9）：`Tracking`/`ControlLaw` 虽"不同模态用法不同"，但选哪个由外部流程切，自身不读状态机，故归算法。
> `ControlLaw` 不在 `step()` 顶层挂，被 `Tracking` 组合调用——选实现经 `Tracking` 间接配（实体仍经 config 决定）。
> `通用数学`（坐标变换 / 限幅 / 滤波）是工具函数、**不是 `step` 单元**，不受本族原则约束。

## 2.3 实体挂接、绑定与执行序

- **挂接（mount）**：实体 `init` 按 config 实例化所需单元类作为成员；同族可挂多个实例（如三轴各一），各持各的状态。
- **绑定（bind）**：实体 `init` 实例化**一份黑板**，把每个单元的 In/Out 视图指针**指向黑板对应槽**。这一步**就是"实体选实现 + 接线"**——选哪个 impl、它的 In/Out 接哪些槽，都在此定。
  - **单写者校验**：每个黑板槽**至多一个单元的 Out 指向它**（多读者 OK），避免覆盖。
  - **反馈 / 保持槽播种**：`航段` / `待飞距` / `编队模式` 等被 t-1 读取的槽，以及 `主机运动状态` / `队形指令` 等**跨实体保持槽**（由广播驱动），`init` 给**合理初值**（t=0 / 首拍无广播时不至读到垃圾）。
- **执行**：`step()` 按**固定拓扑序**逐个 `unit.step()`，单元经已绑定视图读写黑板。本轮单模态、单条固定 DAG。
  - **通信受扰语义（首拍无广播 / 丢包）**：`主机运动状态 / 编队模式 / 队形指令` 由 `收消息` 写、黑板跨拍保持；无新广播时 `收消息` 不覆写 → **沿用上次有效值（hold-last-valid）**，首拍用 `init` 播种初值。本轮**不引入新鲜度 / 降级**（见 §2.5 TODO）。
  - **反馈（t-1）= 单槽 + 读写顺序**：单槽既存本拍又存上拍——靠"**先读后写 + 固定顺序**"取到 t-1（`轨迹规划` 在 `航段` 槽先读 t-1 再写 t；`待飞距` 因 `轨迹规划` 排在 `位置解算` 前、读到的自然是上拍值）。**本轮不双缓冲**；执行顺序成为正确性不变式、不可乱动（双缓冲留到"动态管线重连"，见 §2.5）。
- **遍历**：实体持单元集合，`reset` / `read_state` 泛型遍历。

```python
# 实体 step() 示意（统一 step(self) 模板；u/y 已在 init 绑到黑板；具体见 5-用例）
def step(self, ctx):
    self._ingest(ctx)              # self_state / mission_command 写入黑板槽（inbox 不入黑板）
    self.rx.step(ctx.inbox)        # 收消息：唯一例外，吃 ctx.inbox(用完即弃)→写黑板（仅挂收消息的实体）
    for unit in self._rest:        # 其余单元固定拓扑序，统一 step(self) 经黑板
        unit.step()                # 含发消息：写 outbox 边界槽
    return self._emit()            # 从黑板取 control / outbox 槽 / status 组装 Output
```

## 2.4 各单元的 u / y（本轮领航-跟随，黑板视图）

> 每个单元的 `u`/`y` = 指向黑板槽的视图；下表给**长机 / 僚机各实现实际读写的槽**（**输入下沉、不拉齐**：长机位置解算不读 `队形/主机`、僚机不读 `航段`）。
> own / host：`位置解算` + `发消息` 用 **主机运动状态**(host=长机；长机自身 host==own)；`轨迹规划` + `误差解算` 用 **本体运动状态**(own)。
> 收 / 发消息的跨实体载荷仍是 `list[MessageEnvelope]`。

| 单元（族） | 库 | 角色 | `u`（读黑板槽 / 注入） | `y`（写黑板槽 / 产出） |
| --- | --- | --- | --- | --- |
| 收消息（`Inbound`） | 流程 | 僚机 | `inbox: list[MessageEnvelope]`（例外签名 `step(self, inbox)`，用完即弃、不入黑板） | `主机运动状态`(=长机运动状态)、`编队模式`、`队形指令` |
| | | 长机 | —（本轮长机不收） | — |
| 任务编排（`Orchestrate`） | 流程 | 长机 | `遥控指令`(=`mission_command`)、`上一拍编队模式` | `编队模式`、`队形指令` |
| | | 僚机 | —（无编排，模态来自收消息） | — |
| 轨迹规划（`TrajectoryPlan`） | 流程 | 长机·航线推进 | `编队模式`、`上一拍航段`、`本体运动状态`、`上一拍待飞距` | `航段` |
| | | 僚机·空规划（预埋） | 同上 | `航段`（本场景为空） |
| 位置解算（`PositionSolve`） | 算法 | 长机·航线插值 | `航段`、`本体运动状态` | `目标指令`、`待飞距` |
| | | 僚机·槽位几何 | `队形指令`、`主机运动状态`；init`(飞机ID, 队形信息)` | `目标指令`（`待飞距` 不写） |
| 误差解算（`DeviationCalc`） | 算法 | 长 / 僚 | `目标指令`、`本体运动状态` | `三轴误差` |
| 跟踪（`Tracking`） | 算法 | 长 / 僚 | `三轴误差` | `三轴加速度`(=`control`) |
| 控制算法（`ControlLaw`） | 算法 | — | 控制误差（被 `跟踪` 组合内部调） | 控制量 |
| 发消息（`Outbound`） | 流程 | 长机 | `队形指令`、`主机运动状态`、`编队模式`、`time_s`；init`(网络拓扑)` | 写黑板 `outbox` 边界槽（`list[MessageEnvelope]`；`_emit` 取出进 `Output.outbox`） |
| | | 僚机 | —（不发） | —（空列表） |

补充：

- **`待飞距` 是反馈槽**：`位置解算`(长机) 写、`轨迹规划`(下一拍) 读；僚机槽位几何不写（输出下沉，下游 `轨迹规划` 本场景为空规划、不读它）。
- **长机 `本体运动状态 == 主机运动状态`**（都 = 自己导航）；留两个槽是为族统一，不是长机真有两个态。僚机 `主机运动状态` 来自 `收消息` 解析的长机广播（t-1）。
- 僚机 `任务编排`/`发消息`、长机 `收消息` 在本场景为空（占位/不参与），但保留族位以备扩展。

### 2.4.1 黑板叶类型（L4）定义

| 叶类型（黑板槽） | 字段 | 说明 |
| --- | --- | --- |
| `运动状态`（NavState） | 见 §1.2.1 | 本体 / 主机两种引用共用此类型 |
| `航段`（Segment） | `航段号`、`起点 ENU{x_m,y_m,altitude_m}`、`终点 ENU`、`速度指令` | 长机轨迹规划产；僚机为空 |
| `目标指令`（Target） | `ENU{x_m,y_m,altitude_m}`、`航迹偏航角 ψ`、`航迹倾角 θ`、`速度 V` | **收敛契约**：长 / 僚机位置解算都产同一类型，下游一致 |
| `三轴误差`（Deviation） | `三轴位置误差{ENU}`、`速度误差{ENU 矢量}` | 长机航线插值时**前向位置误差为 0**（目标点投影在航段上） |
| `编队模式`（Mode） | `phase: ModeEnum` | `{集结, 保持, 重构}`，本轮恒 `保持`；留壳容纳将来各模态参数 |
| `队形指令` | 队形类型标签（`三角`/`wedge`/`line`/`echelon`） | 编排产、广播下发；僚机用它 + `飞机ID` 在 `队形信息` 里查自己的相对偏置 |
| `待飞距` | `float` | 位置解算（航线插值）产、回喂轨迹规划的反馈量 |
| `ModeSource` | `编队模式`(上一拍)、`mission_command?` | 编排入参；本轮仅长机用 |
| `队形信息`（共有静态） | `飞机ID → 相对偏置` 几何表 | `init` 注入（算法库共享）；粒度 = 飞机 id 级 |
| `网络拓扑`（共有静态） | 通信拓扑 | `init` 注入 |
| `AlgorithmStatus` | 当前模态、关键误差摘要 | 供 `Output.status`；字段随实现补 |

> **`ParsedInbox` 不再是单独传递的结构体**：`收消息` 直接把 `主机运动状态 / 编队模式 / 队形指令` 写进黑板对应槽，下游各读各的。
> **`inbox` 不入黑板、`outbox` 是边界槽**：`inbox` 用完即弃（`收消息` 例外签名 `step(self, inbox)`）；`outbox` 是黑板边界槽（`发消息` 写、`_emit` 取出进 `Output.outbox`）。
> **保持槽语义（通信受扰）**：`主机运动状态 / 编队模式 / 队形指令` 由广播驱动、黑板跨拍保持；无新广播时沿用上次有效值、首拍用 `init` 播种初值（hold-last-valid，见 §2.3）。本轮不引入新鲜度 / 降级。
> **粒度 = 飞机 id 级**：广播只发 `队形指令 + 长机运动状态 + 编队模式`，槽位映射收进静态 `队形信息`（按 `飞机ID` 索引），不发 `slot_assignment`。
