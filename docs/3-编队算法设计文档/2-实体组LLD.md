# 实体组 LLD

> 实体组是编队算法中**唯一持有状态**的模块：每个实体 = 一个对象，实例化并组合它需要的算法库 / 流程库单元，持有这些子对象即持有全部维护数据。
> 接口契约见 `1-LLD综述.md`；本册讲实现方法。

## 1. 实体类型

| 实体 | `entity_type` | 组成 | 产 control |
| --- | --- | --- | --- |
| 飞机本体 | `flight` | 飞行单元；长机额外寄宿协调单元 | 是 |
| 协调本体（将来：地面站 / 虚拟节点 / 参考节点） | `coordination` | 仅协调单元，不飞 | 否 |

> 本轮只有飞机本体：长机本体（飞行 + 协调）、僚机本体 ×N（飞行）。

## 2. 挂接 / 实例化 / 绑定

- `init(entity_id, config)`：按 `config` **实例化所需的库单元类并组合进实体**（挂接 = 把单元对象作为实体的成员）；静态参数（增益、队形几何表 `队形信息`、飞机ID、网络拓扑、`entity_type`）在此注入各单元的构造函数。
- **挂接 = 为每个族选一个实现**：每个单元都是一个策略族（抽象基类 + 多实现，见 `1-LLD综述.md` §2.2）；实体挂接时**给每个族挑定一个实现类并实例化**（如位置解算选"航线插值"还是"槽位几何"、编排选"常量保持"还是"状态机"、发选"广播"还是"不发"）。**"由实体来选实现"就落在这一步**——情景差异靠选不同实现，**不靠单元内 `if/else`**。
- **绑定 = 实例化黑板 + 把各单元 In/Out 指针指向黑板槽**（见 `1-LLD综述.md` §2.0/§2.3）：`init` 建**一份 per-entity 黑板**，把每个单元（已选定的 impl）的输入 / 输出视图指针绑到对应黑板槽。这一步顺带两件校验 / 准备：
  - **单写者**：每个黑板槽至多一个单元的 Out 指向它（多读者 OK），避免覆盖；
  - **反馈槽播种**：`航段 / 待飞距 / 编队模式` 等被 t-1 读取的槽给合理初值。
- **可重入靠实例化**：N 个僚机 = `Wingman` 类的 N 个实例，**各持各的黑板与单元**；库类定义是那份共享代码，禁止全局 / 类级可变态（黑板与视图均 per-instance）。
- **C 移植**：实体 / 单元对象 ≡ `struct + 接收 struct 指针的函数`；黑板 ≡ 结构体、单元 In/Out ≡ 含指针的结构体；挂接 ≡ 工厂创建，绑定 ≡ 把指针指向黑板成员，所有权归实体。

## 3. `step` 编排（本轮：黑板 + 固定拓扑序）

`step()` 先把注入量（`self_state / inbox / mission_command`）写进黑板对应槽，再按**固定拓扑序**逐个 `unit.step()`——单元经 `init` 已绑定的 In/Out 视图读写黑板，**实体不每拍组装 `u`**（绑定在 `init` 一次完成）；单元自身状态（PID 积分等）留在各自实例里。本轮单模态、单条静态 DAG。反馈槽（`航段 / 待飞距 / 编队模式`）靠"先读后写 + 固定顺序"取 t-1，本轮不双缓冲（见 `1-LLD综述.md` §2.3）。

```python
# 僚机本体（无任务编排 / 发消息；u/y 已在 init 绑到黑板）
def step(self, ctx: FormationAlgorithmContext) -> FormationAlgorithmOutput:
    self._ingest(ctx)              # self_state→本体运动状态、inbox 写入黑板槽
    self.rx.step()                 # 收消息(流程)：inbox → 主机运动状态, 编队模式, 队形指令
    self.planner.step()            # 轨迹规划(流程, 空规划预埋)：→ 航段(空)
    self.possolve.step()           # 位置解算(算法, 槽位几何)：队形指令+主机运动状态 → 目标指令
    self.devcalc.step()            # 误差解算(算法)：目标指令+本体运动状态 → 三轴误差
    self.tracker.step()            # 跟踪(算法, 内部调控制算法)：三轴误差 → 三轴加速度
    return self._emit()            # 从黑板取 三轴加速度→control、outbox=[]、status
```

长机本体在**一个对象**内同时跑飞行流与协调流，共享**同一份黑板**（详见 `5-用例-领航跟随保持.md` 的长机 `step`）。

## 4. 生命周期

| 接口 | 语义 |
| --- | --- |
| `init` | 实例化并挂接单元，注入静态参数 |
| `step` | 一拍：消费注入的 `ctx`，编排单元，返回 `{control?, outbox, status}` |
| `read_state` | 汇出实体（及各子单元）状态，供落盘 / 快照（散落在子对象，需统一暴露） |
| `reset` | 回到初始：**整实体重新实例化**（最省事，与 `1-仿真控制HLD` 的 `reset() 重建内部模块` 一致），或逐单元 `reset` |
| `close` | 释放资源 |
