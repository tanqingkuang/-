# 一、说明

本文档描述编队算法“流程组”各单元的低层设计（LLD），供人阅读，也用于指导代码开发。

流程组属于 HLD 四层中的 L2，与算法组的区别是：**流程组感知 Mode（编队模态）**——它产生、使用、切换模态，决定“做什么、选哪个”，再把具体计算交给算法组。每个单元同样是一个策略族：一个抽象类加若干子类。

约定（与算法组一致）：

- 接口与命名遵循《0-HLD.md》2.6；流程组为 L2，接口为 `init/step/reset`（无 `close`），`step` 的输出 `y` 原地写出。
- 数据通过黑板 `Context` 交换；输入/输出结构体字段是端口，运行期由对象组绑定。**边界 I/O（`inbox`/`outbox`/`remote`）不在 `Context` 内**，由对象组在边界持有（见《0-HLD.md》3.3）；本文对这类端口注明“边界，由对象组持有”。
- 叶类型见《1-Context和叶类型.md》。`MessageEnvelope` 为通信报文类型，定义见《5-1-通信功能LLD.md》§4.1（`src/common/envelope.py`），本文直接引用。
- 覆盖用例：1 个航段 + 集中式通信 + 长机僚机 + PID 控制 + 三角队形。

流程组单元清单：

| 单元 | 名称(XXX) | 作用 |
|---|---|---|
| 收消息 | Inbound | 接收并解析其它实体发来的报文，把其中数据写入 Context |
| 发消息 | Outbound | 把本机要广播的数据打包成报文 |
| 任务编排 | FormationTask | 根据遥控指令与当前状态，决定编队模态与队形 |
| 轨迹规划 | TraPlan | 根据模态与当前状态，给出要跟踪的航段 |

# 二、各流程

## 2.1 收消息 Inbound

作用：从 `inbox`（收到的报文列表）解析出需要的数据，写入 `Context`。领航跟随保持场景中，僚机用它接收长机广播；长机不收消息。

### 2.1.1 抽象类

```python
class InboundBase:
    def init(self, cfg: "InboundInitS") -> None: ...
    def step(self, u: "InboundInputS", y: "InboundOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class InboundInitS:
    pass

@dataclass
class InboundInputS:
    inbox: list[MessageEnvelope] = field(default_factory=list)  # 边界输入，由对象组持有；收到的报文列表

@dataclass
class InboundOutputS:
    leaderState: MotionProfS = None   # 端口 → Context.leaderState，解析出的长机运动状态
    cmd: FormSnapshotS = None         # 端口 → Context.cmd，解析出的编队模式与队形
```

### 2.1.2 子类：长机广播接收 RallyLeaderFollower（统一入站）

实现方法：从 `inbox` 中找到长机广播的报文，解析出长机实际运动状态写入 `y.leaderState`，编队模式与队形写入 `y.cmd`，解析 `cmd["leader"]`（长机有效运动指令）写入 `y.leaderCmd`，并解析公共到达时刻与圈数计划。旧格式报文没有 `cmd["leader"]` 时，`leaderCmd` 回退写入 `leader_state`，保证接收端仍有可用参考帧。断链时 `inbox` 为空，沿用上一帧有效数据，不报错。

结构体：输入沿用 `InboundInputS`；输出绑定共享黑板的 `leaderState/leaderCmd/cmd/rallyPlan`。

接口：实现 `init/step/reset`。

测试用例设计：

- 给定一条合法长机报文：`leaderState`、`leaderCmd`、`cmd` 和公共计划解析正确。
- 旧格式报文缺少 `cmd["leader"]`：`leaderCmd` 回退为 `leaderState`。
- `inbox` 为空（断链）：输出保持上一帧值。
- `inbox` 含多条报文：能筛出长机报文。

### 2.1.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| RallyLeaderFollower 长机广播接收 | init/step/reset | InboundInputS：inbox(list[MessageEnvelope]，边界) | 共享黑板：leaderState、leaderCmd、cmd、rallyPlan | 僚机接收并解析长机广播；`leader_state` 写入实际长机状态，`cmd["leader"]` 写入有效长机指令供 SlotGeometry 建系，缺失时回退 leaderState；普通保持与集结共用；断链时沿用上一帧 |

## 2.2 发消息 Outbound

作用：把本机要广播的数据打包成报文列表（`outbox`）。领航跟随保持场景中，长机用它广播自身状态与队形指令；僚机不发消息。

### 2.2.1 抽象类

```python
class OutboundBase:
    def init(self, cfg: "OutboundInitS") -> None: ...
    def step(self, u: "OutboundInputS", y: "OutboundOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class OutboundInitS:
    selfId: str = ""                                      # 本机 ID（节点ID，如 "A01"），作为报文 source
    netWork: list[NetWorkS] = field(default_factory=list) # 网络拓扑（多条链路），决定本机能发给谁；init 时注入，存于 self

@dataclass
class OutboundInputS:
    cmd: FormSnapshotS = None         # 端口 → Context.cmd，编队模式与队形
    selfState: MotionProfS = None     # 端口 → Context.selfState，本机（长机）运动状态

@dataclass
class OutboundOutputS:
    outbox: list[MessageEnvelope] = field(default_factory=list)  # 边界输出，由对象组持有
```

### 2.2.2 子类：长机广播 RallyLeaderBroadcast（统一出站）

实现方法：

1. 从 `self.netWork` 求本机能发到的接收方：取所有 `startId == self.selfId` 链路的 `endId`；再加上所有 `endId == self.selfId` 且 `dir == DUPLEX` 链路的 `startId`（双向链路反向也可达）。去重得到接收方 ID 列表。
2. 把长机实际运动状态、有效运动指令、编队命令和公共到达计划打包成 `MessageEnvelope` 的 payload，`source = self.selfId`，`target = 接收方 ID 列表`。有效运动指令放在 `payload["cmd"]["leader"]`，供僚机建立槽位坐标系。
3. `outbox = [该 envelope]`。

注意：多接收方由 `MessageEnvelope.target`（支持单播/多播/广播）承载，通信模块按 target 扇出投递；本单元只决定“发给谁”，不复制多份报文（通信功能见《5-1-通信功能LLD.md》）。

结构体：初始化与输出沿用 `OutboundInitS/OutboundOutputS`；输入绑定共享黑板的 `cmd/selfState/effectiveCmd/rallyPlan`。

接口：实现 `init/step/reset`。

测试用例设计：

- 给定长机状态、有效指令、编队命令和公共计划：envelope 的 payload 字段正确，`source` 为本机 ID。
- 未绑定 `leaderCmd`：`payload["cmd"]["leader"]` 回退为 `selfState`，兼容旧调用方。
- 拓扑中本机有多条出边：`target` 覆盖全部接收方。
- 双向链路反向（`endId==本机` 且 `dir==DUPLEX`）：对端也计入接收方。
- 本机无出边：`outbox` 为空。

### 2.2.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| RallyLeaderBroadcast 长机广播 | init/step/reset | 共享黑板：cmd、selfState、effectiveCmd、rallyPlan；配置 OutboundInitS：selfId、netWork(list[NetWorkS]) | OutboundOutputS：outbox(list[MessageEnvelope]，边界) | 长机据拓扑筛出接收方，打包一条 envelope（target 多播/广播）发出；`leader_state` 携带实际状态，`cmd["leader"]` 携带有效指令；普通保持与集结共用 |

## 2.3 任务编排 FormationTask

作用：根据遥控指令和上一拍模态，决定本拍的编队模态与队形，写入 `Context.cmd`。领航跟随保持场景中，长机做编排；僚机不编排（模态来自长机广播）。

### 2.3.1 抽象类

```python
class FormationTaskBase:
    def init(self, cfg: "FormationTaskInitS") -> None: ...
    def step(self, u: "FormationTaskInputS", y: "FormationTaskOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class FormationTaskInitS:
    pass

@dataclass
class FormationTaskInputS:
    remote: RemoteCmdS = None    # 边界输入，由对象组持有；遥控指令
    cmd: FormSnapshotS = None    # 端口 → Context.cmd，上一拍编队模态（读旧值）

@dataclass
class FormationTaskOutputS:
    cmd: FormSnapshotS = None    # 端口 → Context.cmd，本拍编队模态与队形（写新值）
```

> 说明：输入的 `cmd`（上一拍）与输出的 `cmd`（本拍）绑定到同一个 `Context.cmd`。因为 `Context` 跨拍常驻、原地迭代，本拍读到的就是上一拍写入的值（参见《1-Context和叶类型.md》二）。

### 2.3.2 子类：通用保持/集结 Rally

实现方法：保持与集结共用 `Rally`。`RallyTaskInitS.enabled=False` 时不进入集结状态机：`remote=NONE` 输出 `NONE`，其余远控阶段统一输出 `HOLD`；`enabled=True` 时按 STANDBY、JOINING、CATCHUP、LOOSE、HOLD 推进集结。目标队形索引可经 `set_pattern_index(index)` 热切换，下一拍随长机广播下发。

结构体：`RallyTaskInitS(FormationTaskInitS)` 包含集结参数，并以 `enabled` 控制是否启用集结能力。`passive` 由实体身份固定：长机为主动模式，僚机为被动模式，调用方无需设置。

接口：实现 `init/step/reset`，另提供 `set_pattern_index(index)` 供运行时热切换；`reset` 把目标索引复位到初始值。

测试用例设计：

- 禁用集结时：`NONE` 保持复位语义，非 `NONE` 指令直接进入 `HOLD`，且不校验盘旋参数。
- 启用集结时：按集结状态机推进；僚机保留入站长机指令，不被本地默认远控覆盖。
- `set_pattern_index(i)` 后下一拍 `cmd.pattern == i`；`reset` 后回到目标初值。

### 2.3.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| Rally 通用保持/集结 | init/step/reset/set_pattern_index | FormationTaskInputS：remote(RemoteCmdS，边界)、cmd(FormSnapshotS)；配置 RallyTaskInitS(enabled/passive/集结参数) | FormationTaskOutputS：cmd(FormSnapshotS) | 禁用时直接保持，启用时执行集结状态机；支持运行时热切换 |

## 2.4 轨迹规划 TraPlan

作用：根据编队模态和当前状态，给出本机要跟踪的航段，写入 `Context.wayLine`。领航跟随保持场景中，长机使用航线规划，僚机使用空策略维持流水线结构一致。

### 2.4.1 抽象类

```python
class TraPlanBase:
    def init(self, cfg: "TraPlanInitS") -> None: ...
    def step(self, u: "TraPlanInputS", y: "TraPlanOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class TraPlanInitS:
    pass

@dataclass
class TraPlanInputS:
    cmd: FormSnapshotS = None      # 端口 → Context.cmd，编队模态
    wayLine: WayLineS = None       # 端口 → Context.wayLine，上一拍航段（读旧值）
    selfState: MotionProfS = None  # 端口 → Context.selfState，本体运动状态

@dataclass
class TraPlanOutputS:
    wayLine: WayLineS = None       # 端口 → Context.wayLine，本拍航段（写新值）
```

### 2.4.2 子类：长机航线 LeaderRoute

实现方法：仿真控制从配置文件读取 `route`，构造 `RouteS`，经 `EntityInitS.route` 注入长机实体；`LeaderRoute.init` 接收整条航线并存于 `self`。若未注入航线，则使用默认单航段航线。`step` 每拍从航线中选出当前 `WayLineS`，写入 `y.wayLine`。输入 `wayLine` 与输出 `wayLine` 绑定同一 `Context.wayLine`，原地迭代。

结构体：

```python
@dataclass
class LeaderRouteInitS(TraPlanInitS):
    route: RouteS = None
```

接口：实现 `init/step/reset`。

测试用例设计：

- 初始：输出为 `init` 注入航线的当前航段。
- 多拍：根据本机位置推进当前航段。

### 2.4.3 子类：空策略 Noop（僚机用）

实现方法：本场景僚机不需要轨迹规划。该子类 `step` 不读写 `Context.wayLine`，用于不需要本机轨迹规划的实体，并保持流水线结构一致。

测试用例设计：`step` 调用后 `Context.wayLine` 不变。

### 2.4.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| LeaderRoute 长机航线 | init/step/reset | TraPlanInputS：cmd(FormSnapshotS)、wayLine(WayLineS)、selfState(MotionProfS) | TraPlanOutputS：wayLine(WayLineS) | 长机持有 init 注入的 RouteS，每拍写出当前航段 |
| Noop 空策略 | init/step/reset | 同上（不使用） | 同上（不写出） | 不读写航段，保持流水线结构一致 |
