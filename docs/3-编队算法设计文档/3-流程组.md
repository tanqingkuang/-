# 一、说明

本文档描述编队算法“流程组”各单元的低层设计（LLD），供人阅读，也用于指导代码开发。

流程组属于 HLD 四层中的 L2，与算法组的区别是：**流程组感知 Mode（编队模态）**——它产生、使用、切换模态，决定“做什么、选哪个”，再把具体计算交给算法组。每个单元同样是一个策略族：一个抽象类加若干子类。

约定（与算法组一致）：

- 接口与命名遵循《0-HLD.md》2.6；流程组为 L2，接口为 `init/step/reset`（无 `close`），`step` 的输出 `y` 原地写出。
- 数据通过黑板 `Context` 交换；输入/输出结构体字段是端口，运行期由对象组绑定。**边界 I/O（`inbox`/`outbox`/`remote`）不在 `Context` 内**，由对象组在边界持有（见《0-HLD.md》3.3）；本文对这类端口注明“边界，由对象组持有”。
- 叶类型见《1-Context和叶类型.md》。`MessageEnvelope` 为通信报文类型，定义见《5-1-通信功能LLD.md》§4.1（`src/common/envelope.py`），本文直接引用。
- 本轮覆盖用例：1 个航段 + 集中式通信 + 长机僚机 + PID 控制 + 三角队形。

流程组单元清单：

| 单元 | 名称(XXX) | 作用 |
|---|---|---|
| 收消息 | Inbound | 接收并解析其它实体发来的报文，把其中数据写入 Context |
| 发消息 | Outbound | 把本机要广播的数据打包成报文 |
| 任务编排 | FormationTask | 根据遥控指令与当前状态，决定编队模态与队形 |
| 轨迹规划 | TraPlan | 根据模态与当前状态，给出要跟踪的航段 |

# 二、各流程

## 2.1 收消息 Inbound

作用：从 `inbox`（收到的报文列表）解析出需要的数据，写入 `Context`。本轮中僚机用它接收长机广播；长机不收消息。

### 2.1.1 抽象类

```python
class InboundBase:
    def init(self, cfg: "InboundInitS") -> None: ...
    def step(self, u: "InboundInputS", y: "InboundOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class InboundInitS:
    pass

@dataclass
class InboundInputS:
    inbox: list[MessageEnvelope] = field(default_factory=list)  # 边界输入，由对象组持有；收到的报文列表

@dataclass
class InboundOutputS:
    leaderState: MotionProfS = None   # 端口 → Context.leaderState，解析出的长机运动状态
    cmd: FormSnapshotS = None         # 端口 → Context.cmd，解析出的编队模式与队形
```

### 2.1.2 子类：长机广播接收 LeaderFollower

实现方法：从 `inbox` 中找到长机广播的报文，解析出长机运动状态写入 `y.leaderState`，编队模式与队形写入 `y.cmd`。断链时 `inbox` 为空：保持上一帧的 `leaderState`/`cmd` 不变（沿用最后一帧），不报错（断链语义详见本单元后续扩展）。

结构体：沿用抽象类，无扩展。

接口：实现 `init/step/reset`。

测试用例设计：

- 给定一条合法长机报文：`leaderState`、`cmd` 解析正确。
- `inbox` 为空（断链）：输出保持上一帧值。
- `inbox` 含多条报文：能筛出长机报文。

### 2.1.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| LeaderFollower 长机广播接收 | init/step/reset | InboundInputS：inbox(list[MessageEnvelope]，边界) | InboundOutputS：leaderState(MotionProfS)、cmd(FormSnapshotS) | 僚机接收并解析长机广播；断链时沿用上一帧 |

## 2.2 发消息 Outbound

作用：把本机要广播的数据打包成报文列表（`outbox`）。本轮中长机用它广播自身状态与队形指令；僚机不发消息。

### 2.2.1 抽象类

```python
class OutboundBase:
    def init(self, cfg: "OutboundInitS") -> None: ...
    def step(self, u: "OutboundInputS", y: "OutboundOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class OutboundInitS:
    selfId: str = ""                                      # 本机 ID（节点ID，如 "A01"），作为报文 source
    netWork: list[NetWorkS] = field(default_factory=list) # 网络拓扑（多条链路），决定本机能发给谁；init 时注入，存于 self

@dataclass
class OutboundInputS:
    cmd: FormSnapshotS = None         # 端口 → Context.cmd，编队模式与队形
    selfState: MotionProfS = None     # 端口 → Context.selfState，本机（长机）运动状态

@dataclass
class OutboundOutputS:
    outbox: list[MessageEnvelope] = field(default_factory=list)  # 边界输出，由对象组持有
```

### 2.2.2 子类：长机广播 LeaderBroadcast

实现方法：

1. 从 `self.netWork` 求本机能发到的接收方：取所有 `startId == self.selfId` 链路的 `endId`；再加上所有 `endId == self.selfId` 且 `dir == DUPLEX` 链路的 `startId`（双向链路反向也可达）。去重得到接收方 ID 列表。
2. 把 `u.selfState`（长机运动状态）和 `u.cmd`（编队模式、队形）打包成 `MessageEnvelope` 的 payload，`source = self.selfId`，`target = 接收方 ID 列表`（集中式下即全部僚机；也可用 `"broadcast"`）。
3. `outbox = [该 envelope]`。

注意：多接收方由 `MessageEnvelope.target`（支持单播/多播/广播）承载，通信模块按 target 扇出投递；本单元只决定“发给谁”，不复制多份报文（通信功能见《5-1-通信功能LLD.md》）。

结构体：沿用抽象类，无扩展。

接口：实现 `init/step/reset`。

测试用例设计：

- 给定 `selfState`、`cmd`：envelope 的 payload 含正确的长机状态与队形，`source` 为本机 ID。
- 拓扑中本机有多条出边：`target` 覆盖全部接收方。
- 双向链路反向（`endId==本机` 且 `dir==DUPLEX`）：对端也计入接收方。
- 本机无出边：`outbox` 为空。

### 2.2.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| LeaderBroadcast 长机广播 | init/step/reset | 输入 OutboundInputS：cmd(FormSnapshotS)、selfState(MotionProfS)；配置 OutboundInitS：selfId、netWork(list[NetWorkS]) | OutboundOutputS：outbox(list[MessageEnvelope]，边界) | 长机据拓扑筛出接收方，打包一条 envelope（target 多播/广播）发出 |

## 2.3 任务编排 FormationTask

作用：根据遥控指令和上一拍模态，决定本拍的编队模态与队形，写入 `Context.cmd`。本轮中长机做编排；僚机不编排（模态来自长机广播）。

### 2.3.1 抽象类

```python
class FormationTaskBase:
    def init(self, cfg: "FormationTaskInitS") -> None: ...
    def step(self, u: "FormationTaskInputS", y: "FormationTaskOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class FormationTaskInitS:
    pass

@dataclass
class FormationTaskInputS:
    remote: RemoteCmdS = None    # 边界输入，由对象组持有；遥控指令
    cmd: FormSnapshotS = None    # 端口 → Context.cmd，上一拍编队模态（读旧值）

@dataclass
class FormationTaskOutputS:
    cmd: FormSnapshotS = None    # 端口 → Context.cmd，本拍编队模态与队形（写新值）
```

> 说明：输入的 `cmd`（上一拍）与输出的 `cmd`（本拍）绑定到同一个 `Context.cmd`。因为 `Context` 跨拍常驻、原地迭代，本拍读到的就是上一拍写入的值（参见《1-Context和叶类型.md》二）。

### 2.3.2 子类：保持 Hold（占位）

实现方法：本场景恒输出“编队保持”模态、三角队形，不做模态切换。集结、重构的切换逻辑在此预埋（按 `remote` 与上一拍模态决定转移），本轮不实现。

结构体：沿用抽象类，无扩展。

接口：实现 `init/step/reset`。

测试用例设计：

- 任意输入：输出模态恒为“编队保持”、队形为三角。
- 预埋项：`remote` 指示切换时的模态转移（本轮仅占位，不作为通过条件）。

### 2.3.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| Hold 保持（占位） | init/step/reset | FormationTaskInputS：remote(RemoteCmdS，边界)、cmd(FormSnapshotS) | FormationTaskOutputS：cmd(FormSnapshotS) | 恒输出“编队保持+三角”；集结/重构预埋未实现 |

## 2.4 轨迹规划 TraPlan

作用：根据编队模态和当前状态，给出本机要跟踪的航段，写入 `Context.wayLine`。本轮中长机做规划；僚机的轨迹规划在本场景无实际意义，仅预埋占位（为兼容后续集结流程）。

### 2.4.1 抽象类

```python
class TraPlanBase:
    def init(self, cfg: "TraPlanInitS") -> None: ...
    def step(self, u: "TraPlanInputS", y: "TraPlanOutputS") -> None: ...
    def reset(self) -> None: ...

@dataclass
class TraPlanInitS:
    pass

@dataclass
class TraPlanInputS:
    cmd: FormSnapshotS = None      # 端口 → Context.cmd，编队模态
    wayLine: WayLineS = None       # 端口 → Context.wayLine，上一拍航段（读旧值）
    selfState: MotionProfS = None  # 端口 → Context.selfState，本体运动状态

@dataclass
class TraPlanOutputS:
    wayLine: WayLineS = None       # 端口 → Context.wayLine，本拍航段（写新值）
```

### 2.4.2 子类：长机航线 LeaderRoute

实现方法：本场景只有 1 个航段，且航段本身不需要验证，故本轮先把该航段以常量形式**写死在本子类内部**（`init` 时构造一个固定的 `WayLineS` 存于 `self`），`TraPlanInitS` 暂不加航段配置字段。`step` 每拍把这个固定航段写入 `y.wayLine`（航段切换逻辑预埋，本轮用不到）。输入 `wayLine` 与输出 `wayLine` 绑定同一 `Context.wayLine`，原地迭代。

> TODO：写死仅为本轮便利，且把场景数据放进了算法单元。后续应改为由 `init` 注入——给 `TraPlanInitS`（或 `EntityInitS`）加航线字段，把航段数据从算法单元里移出去。

结构体：沿用抽象类，无扩展。

接口：实现 `init/step/reset`。

测试用例设计：

- 初始：输出为写死在子类内的那 1 个航段。
- 多拍：航段保持不变（本场景不切段）。

### 2.4.3 子类：占位 Noop（僚机用）

实现方法：本场景僚机不需要轨迹规划。该子类 `step` 不做任何读写（空实现），仅为保持流水线结构一致、兼容后续集结流程而存在。后续若僚机需要真正的轨迹规划，只需新增一个有意义的子类替换 Noop 即可，属于等价扩展，不影响流水线结构与调用顺序。

测试用例设计：`step` 调用后 `Context.wayLine` 不变。

### 2.4.N 整理

| 子类 | 接口 | 输入结构体（叶类型） | 输出结构体（叶类型） | 支持的场景与功能 |
|---|---|---|---|---|
| LeaderRoute 长机航线 | init/step/reset | TraPlanInputS：cmd(FormSnapshotS)、wayLine(WayLineS)、selfState(MotionProfS) | TraPlanOutputS：wayLine(WayLineS) | 长机产出/维持单航段（航段暂写死在子类内，TODO 改 init 注入）；切段预埋 |
| Noop 占位 | init/step/reset | 同上（不使用） | 同上（不写出） | 僚机占位，不做任何事；兼容后续集结 |
