# 一、说明

本文档描述编队算法流程组的当前低层设计。流程组属于 L2，负责通信处理、任务编排和轨迹规划；位置解算与位置跟踪属于算法组。

所有流程遵循同一运行契约：

```python
class ProcessBase:
    def bind(self, runtime: EntityRuntimeS) -> None: ...
    def init(self, cfg: ProcessInitS) -> None: ...
    def step(self) -> None: ...
    def reset(self) -> None: ...
```

- `EntityBase` 固定执行 `FormationInbound → Rally → TraPlanManager → PosCalcManager → PosTrackManager → FormationOutbound`，所有身份顺序一致。
- 每个流程在 `bind(runtime)` 中建立自己的私有输入/输出端口；运行期 `step()` 不接收业务参数。
- `FormContextS` 保存跨拍或跨流程状态；`EntityRuntimeS` 另外持有 `remote/inbox/outbox/PosTrackDiagS` 等边界对象。
- Profile 只决定三个 Manager 的状态路由产品，不改变固定流程类型或顺序。

流程组清单：

| 流程 | 当前容器 | 作用 |
| --- | --- | --- |
| 收消息 | `FormationInbound` | 按 topic 解析编队消息并原子更新黑板 |
| 任务编排 | `Rally` | 主动生成或被动保留任务阶段与队形 |
| 轨迹规划 | `TraPlanManager` | 按 Profile 路由 `LeaderRoute/Noop` 缓存产品 |
| 发消息 | `FormationOutbound` | 按初始化固定的消息产品组包或保持静默 |

# 二、各流程

## 2.1 FormationInbound

### 2.1.1 接口与绑定

```python
class InboundBase:
    def bind(self, runtime: EntityRuntimeS) -> None: ...
    def init(self, cfg: InboundInitS) -> None: ...
    def step(self) -> None: ...
    def reset(self) -> None: ...
```

`FormationInbound.bind()` 将私有输入绑定到 `runtime.inbox/context.clock`，将私有输出绑定到 `context.leaderState/leaderCmd/cmd/rallyPlan/followerStates`。所有长机、僚机都执行该流程；没有匹配消息时不修改黑板。

### 2.1.2 消息路由

- `formation.leader`：调用长机广播解析器，全部字段合法后原子更新 `leaderState/leaderCmd/cmd/rallyPlan`；缺少本机圈数时计划不生效。
- `formation.follower_status`：解析僚机状态，按节点 ID 原地更新 `followerStates`；非法消息不得刷新超时时间。
- 未知 topic：直接忽略。
- 同一拍同一节点多条有效消息：按 inbox 顺序处理，最后一条生效。

`FormationInbound` 不感知实体角色和任务阶段，旧 `RallyLeaderFollower/FollowerStatus` 仅保留协议解析辅助能力，不再作为实体流程容器分别装配。

## 2.2 FormationOutbound

### 2.2.1 接口与固定产品

```python
class OutboundBase:
    def bind(self, runtime: EntityRuntimeS) -> None: ...
    def init(self, cfg: OutboundInitS) -> None: ...
    def step(self) -> None: ...
    def reset(self) -> None: ...
```

`FormationOutbound.bind()` 将输入绑定到 `cmd/selfState/selfCmd/effectiveCmd/rallyPlan/posCalcStatus`，输出绑定到 `runtime.outbox`。`init()` 根据 `FormationOutboundInitS.messageType` 固定组包函数，运行期不再判断角色：

| `OutboundMessageE` | 行为 |
| --- | --- |
| `NOOP` | 清空并保持空 outbox；普通 HOLD 僚机使用 |
| `LEADER_BROADCAST` | 广播长机状态、有效指令、任务命令和公共计划 |
| `FOLLOWER_STATUS` | 向显式 `leaderId` 回报位置误差和 `PosCalcStatus` |

每拍先清空 outbox，再调用固定 writer。长机目标由通信拓扑解析；僚机状态回报必须提供 `leaderId`。

## 2.3 Rally

### 2.3.1 接口与身份模式

```python
class FormationTaskBase:
    def bind(self, runtime: EntityRuntimeS) -> None: ...
    def init(self, cfg: FormationTaskInitS) -> None: ...
    def step(self) -> None: ...
    def reset(self) -> None: ...
```

保持与集结共用 `Rally`：

- `enabled=False`：`remote=NONE` 输出 NONE，其他远控阶段直接输出 HOLD，不创建集结生命周期。
- 长机实体固定 `passive=False`，由本地 remote 和僚机回报主动推进状态机。
- 僚机实体固定 `passive=True`，统一入站先写入长机命令，Rally 随后保留该命令；仅本地 STANDBY 可优先覆盖。
- `set_pattern_index()` 热切换目标队形；`reset()` 恢复初始索引并清除本轮计划。

### 2.3.2 集结状态机

```text
STANDBY → RALLY/JOINING → RALLY/CATCHUP → RALLY/LOOSE → HOLD
```

- JOINING：收齐有效基础航程，生成一次性公共时刻和圈数计划，等待全员切出。
- CATCHUP：全部期望僚机同时满足位置和航向门限并稳定指定时间后进入 LOOSE。
- LOOSE：先要求全部期望僚机在 `convergenceRadius_m` 内连续稳定；最终进入 HOLD 和输出 `rallyCompleted=True` 还必须全部满足 `tightRadius_m`。
- 不存在 `COMPRESS`、运行期槽位缩放或缩放速度前馈。

`rallyCompleted` 仅在自然完成的转换拍为 True；外部强制 HOLD 不产生完成事件。

## 2.4 TraPlanManager

### 2.4.1 Manager 与产品接口

`TraPlanManager` 在初始化时遍历实体 Profile 的完整路由表，只创建实际用到的缓存产品；运行期按 `(cmd.stage, cmd.step)` 严格查表并调用产品的无参 `step()`。

```python
class TraPlanBase:
    def bind(self, runtime: EntityRuntimeS) -> None: ...
    def init(self, cfg: TraPlanInitS) -> None: ...
    def step(self) -> None: ...
    def reset(self) -> None: ...
```

| 产品 | 行为 |
| --- | --- |
| `LeaderRoute` | 持有初始化航线，根据本机状态推进当前/下一航段并写入黑板 |
| `Noop` | 不修改航段；僚机和长机停控/JOINING 状态使用 |

有状态产品只在初始化期创建一次，阶段切换只更换缓存引用；`reset()` 复位所有已创建产品但不重建实例。
