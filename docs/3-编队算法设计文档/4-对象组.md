# 一、说明

本文档描述编队算法“对象组”的低层设计（LLD），供人阅读，也用于指导代码开发。

对象组属于 HLD 四层中的 L1（实体）。一个实体 = 一架飞机本地的算法对象。它的职责是**装配**：实例化 `EntityRuntimeS`（其中包含黑板 `Context`、通信边界和诊断对象）、实例化固定六流程，并让每个流程把私有端口绑定到运行时对应对象；每拍 `step` 按固定顺序调用各流程。实体是运行时数据的所有者，流程之间不直接传参。

约定：

- 接口与命名遵循《0-HLD.md》2.6；对象组为 L1，接口为 `init/step/reset/close`，`step` 的输出 `y` 原地写出。
- 边界 I/O 与配置归对象组（见《0-HLD.md》3.3）：`inbox`/`outbox`/`remote` 由 `EntityRuntimeS` 持有；初始化配置（`FormCommInitS`、`FormSelfInitS`、`list[WayPointInputS]`、各单元配置）由实体在 `init` 时注入各单元。
- 各单元的私有端口绑定到 `Context` 或运行时的哪个字段，已在《2-算法组.md》《3-流程组.md》各单元中注明；本文只说明实体如何完成这些绑定与调用顺序。
- 控制诊断量不放入 `Context`，由算法写入 `EntityRuntimeS.posTrackDiag`；对象组每拍汇总到 `EntityOutputS`，再交给仿真控制。
- 叶类型见《1-Context和叶类型.md》。本文档讲设计与实现；验证结果（仿真与分析）见《5-用例-领航跟随保持.md》。

# 二、对象组

## 2.1 接口

```python
class EntityBase:
    def init(self, cfg: "EntityInitS") -> None: ...   # 实例化 Context、各单元与其 I/O、边界 I/O；完成端口绑定
    def step(self, u: "EntityInputS", y: "EntityOutputS") -> None: ...  # 写入边界输入 → 按顺序调用各单元 → 读出边界输出
    def reset(self) -> None: ...      # 复位 Context 与各单元
    def close(self) -> None: ...      # 释放（L1 才有）
```

长机、僚机均由 `EntityBase` 以固定顺序运行 `FormationInbound → Rally → TraPlanManager → PosCalcManager → PosTrackManager → FormationOutbound`。具体实体只负责边界适配、初始化配置及 `Profile` 中的产品选择；角色不再改变一级调用链。

## 2.2 关键结构体

实体的初始化配置、每拍输入、每拍输出：

```python
@dataclass
class EntityInitS:                 # 实体初始化配置（实体持有并分发给各单元）
    selfInit: FormSelfInitS = field(default_factory=FormSelfInitS)   # 本机独有信息（机 ID）
    commInit: FormCommInitS = field(default_factory=FormCommInitS)   # 共有信息（网络拓扑、队形几何）
    route: list[WayPointInputS] = field(default_factory=list)         # 长机原始航点输入；leader.init() 转换为 list[WayLineS]
    control_period_s: float = 0.05                                      # 上层注入的控制算法处理周期，单位 s
    # 各单元的 InitS（如 PID 增益）由实体在此聚合或在 init 内构造，分发给对应单元

@dataclass
class EntityInputS:                # 每拍由平台喂入的边界输入
    selfState: MotionProfS = None                              # 本体运动状态（传感器）
    inbox: list[MessageEnvelope] = field(default_factory=list) # 收到的报文
    remote: RemoteCmdS = None                                  # 遥控指令

@dataclass
class EntityOutputS:               # 每拍交给平台的边界输出
    selfAccCmd: AccInEarthS = None                             # 三轴加速度指令（送控制/执行）
    selfCmd: MotionProfS = None                                # 本机目标运动状态（位置解算结果，东北天）
    controlDiag: PosTrackDiagS = None                          # 控制诊断量（叶类型定义见《1-Context和叶类型.md》）
    outbox: list[MessageEnvelope] = field(default_factory=list)# 待发报文
```

## 2.3 边界 I/O 与配置的归属

按《0-HLD.md》3.3，以下数据不进 `Context`，由实体持有：

- **边界输入** `inbox`、`remote`：实体在 `init` 时各建一个持有对象（`remote` 用 `RemoteCmdS` 包装以满足端口绑定粒度）；收消息/任务编排单元的对应端口绑定到这些实体持有对象。
- **边界输出** `outbox`：实体持有；发消息单元的输出端口绑定到它。
- **配置** `FormCommInitS`、`FormSelfInitS`、`list[WayPointInputS]`、`control_period_s`、各单元 InitS：仿真控制从配置文件读取 `formation.formation_files` 并加载外部队形文件，生成 `commInit.formPat` / `commInit.formPos`，读取 `route` 生成 `list[WayPointInputS]`，按 `step_s * algorithm_decimation` 计算 `control_period_s`，实体在 `init` 时持有并注入各单元（例如注入僚机 `SlotGeometry`，把 `list[WayPointInputS]` 通过 `waypoint_inputs_to_waylines()` 转换后注入长机 `LeaderRoute`，把 `control_period_s` 注入 `PidCompose` 的三轴 PID，把 `commInit.netWork` 与 `selfInit.id` 注入长机 `Outbound`）。

## 2.4 init 与 step 的职责

- `init`：① 实例化 `Context`；② 实例化本实体方式所需的各单元子类及其 InputS/OutputS；③ 实例化边界 I/O 持有对象；④ 调用各单元 `init` 注入配置；⑤ 把各单元 I/O 的每个端口绑定到 `Context` 子对象或边界持有对象（绑定一次，之后不再重赋值）。
- `step`：① 把 `u`（本体状态、inbox、remote）写入对应位置（`Context.selfState`、实体 inbox、实体 remote）；② 按本方式顺序依次调用各单元 `step`；③ 把结果（`Context.selfAccCmd`、`Context.selfCmd`、各单元 `diag`、实体 outbox）读出到 `y`。

数据边界约定：

```text
算法组/流程组
  -> 写各自 OutputS 的 y.xxx 或 y.diag
  -> 对象组绑定到 Context 或对象组持有的诊断对象
  -> EntityOutputS 显式带出 selfAccCmd / selfCmd / controlDiag / outbox
  -> SimCtrl 只读取 EntityOutputS，不直接读取对象组内部 Context
```

其中 `Context` 只保存控制闭环需要跨拍保留或被多个单元读写的业务状态；`controlDiag` 是对本拍控制过程的观测副本，供日志和 UI 使用，不反向参与算法计算。

# 三、各场景

## 3.1 场景一：领航跟随保持

验证场景说明：1 个航段 + 集中式通信 + 长机僚机 + PID 控制 + 三角队形。长机与僚机统一执行固定六步链，身份 Profile 决定各 Manager 使用的产品，部分产品为空策略。两机的唯一耦合点是长机广播的报文经平台搬运进入僚机的 `inbox`（跨拍传递，存在至少 1 拍信息滞后，见《0-HLD.md》4.2）。

保持与集结共用两种通用实体身份：长机 `LeaderEntity`、僚机 `FollowerEntity`。保持场景以 `EntityInitS.rally_enabled=False` 禁用集结能力，不另建 Hold 实体。

### 3.1.1 各单元使用的子类

| 单元 | 长机 LeaderEntity | 僚机 FollowerEntity |
|---|---|---|
| 收消息 Inbound | FormationInbound（无匹配报文时静默） | FormationInbound（解析长机广播） |
| 任务编排 FormationTask | Rally（`enabled=False`，主动身份） | Rally（`enabled=False`，被动身份） |
| 轨迹规划 TraPlan Manager | LeaderRoute（按航线选当前航段） | Noop（空产品） |
| 位置解算 PosCalc Manager | RouteInterp（航线插值） | SlotGeometry（最终槽位几何）；禁用集结时不创建 RallyJoinPos |
| 跟踪 PosTrack Manager | PidCompose（前向 PPI：kpPos=0 的速度环） | PidCompose（前向 PPI：kpPos>0 的位置-速度串级环） |
| 发消息 Outbound | FormationOutbound（长机广播） | FormationOutbound（Noop，保持场景静默） |

### 3.1.2 调用顺序（一拍 step 内）

长机和僚机都由 `EntityBase.step()` 执行唯一顺序：

```
Inbound → FormationTask → TraPlan → PosCalc → PosTrack → Outbound
```

说明：流程容器和顺序完全一致，差异仅在 Profile 选中的产品及初始化参数。长机主动生成模态、推进航线并广播；僚机先解析长机广播，再由被动 Rally 保留入站指令，使用 SlotGeometry 计算最终槽位。两者跟踪同为 PidCompose，仅前向通道增益按角色分化——长机前向只控速度（`kp=ki=0`、速度比例走 `kd`，默认 `kiv=0` 即纯速度 P、`kiv` 预留），僚机前向控待飞距位置（`kiv=0`、位置走 `kp/ki`、`kd` 作阻尼）。

### 3.1.3 端口绑定要点

绑定关系由“各单元选了哪个子类”即可确定（各端口绑到 `Context` 哪个字段见《2-算法组.md》《3-流程组.md》），此处只列两机的差异点：

- 长机：`PosTrack` 写入有效指令；`Outbound` 从共享黑板读取 `cmd/selfState/effectiveCmd/rallyPlan` 并写入实体 `outbox`；`Inbound` 仍在固定链中，未收到关注报文时不修改状态。
- 僚机：`Inbound` 从实体 `inbox` 写入共享黑板的 `leaderState/leaderCmd/cmd/rallyPlan`；`SlotGeometry` 读取长机状态、有效指令和最终槽位；有效指令无水平航迹时回退 `leaderState`。保持配置下 `Outbound` 使用 Noop 产品。

### 3.1.4 场景级联试 / 验证入口

- 联试目标：长机僚机各自实例化、绑定、按上述顺序跑通一拍至多拍，数据沿 `Context` 正确流转；长机 `outbox` 经平台搬运进入僚机 `inbox` 后，僚机能据此跟随。
- 入口：通过 `create_leader_follower_entity(EntityProfileE.LEADER/FOLLOWER)` 构造通用实体，给定 `rally_enabled=False` 的初始配置（机 ID、拓扑、队形几何、控制周期、PID 增益、航线），按拍驱动 `step`，平台在拍间搬运报文。
- 通过判据的方向：僚机目标位置稳定收敛到“长机位置 + 三角槽位偏移”，跟踪误差收敛。
- 具体仿真结果与分析见《5-用例-领航跟随保持.md》。
