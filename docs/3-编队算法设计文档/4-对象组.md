# 一、说明

本文档描述编队算法“对象组”的低层设计（LLD），供人阅读，也用于指导代码开发。

对象组属于 HLD 四层中的 L1（实体）。一个实体 = 一架飞机本地的算法对象。它的职责是**装配**：实例化黑板 `Context`、实例化所需的算法组/流程组单元及其输入输出、把每个端口绑定到 `Context`（或边界）对应的对象，并在每拍 `step` 里按固定顺序调用各单元。实体是“持有全部维护数据”的唯一所有者；单元之间不直接传参，全部经 `Context` 交换。

约定：

- 接口与命名遵循《0-HLD.md》2.6；对象组为 L1，接口为 `init/step/reset/close`，`step` 的输出 `y` 原地写出。
- 边界 I/O 与配置归对象组（见《0-HLD.md》3.3）：`inbox`/`outbox`/`remote` 由实体持有；初始化配置（`FormCommInitS`、`FormSelfInitS`、`RouteS`、各单元配置）由实体在 `init` 时持有，并注入各单元的 `init`。
- 各单元的端口绑定到 `Context` 的哪个字段，已在《2-算法组.md》《3-流程组.md》各单元中注明；本文只说明实体如何完成这些绑定与调用顺序。
- 控制诊断量不放入 `Context`。若某个算法小模块产生只供日志/UI 使用的临时量，应由该模块的 `OutputS.diag` 显式输出；对象组持有诊断对象并在每拍汇总到 `EntityOutputS`，再交给仿真控制。
- 叶类型见《1-Context和叶类型.md》。本文档讲设计与实现；验证结果（仿真与分析）见《5-用例-领航跟随保持.md》。

# 二、对象组

## 2.1 接口

```python
class EntityBase:
    def init(self, cfg: "EntityInitS") -> None: ...   # 实例化 Context、各单元与其 I/O、边界 I/O；完成端口绑定
    def step(self, u: "EntityInputS", y: "EntityOutputS") -> None: ...  # 写入边界输入 → 按顺序调用各单元 → 读出边界输出
    def reset(self) -> None: ...      # 复位 Context 与各单元
    def close(self) -> None: ...      # 释放（L1 才有）
```

每种实体方式（长机、僚机）是 `EntityBase` 的子类，在 `init` 里实例化自己这套单元子类并绑定端口、在 `step` 里按本方式的顺序串联（见第三章）。

## 2.2 关键结构体

实体的初始化配置、每拍输入、每拍输出：

```python
@dataclass
class EntityInitS:                 # 实体初始化配置（实体持有并分发给各单元）
    selfInit: FormSelfInitS = field(default_factory=FormSelfInitS)   # 本机独有信息（机 ID）
    commInit: FormCommInitS = field(default_factory=FormCommInitS)   # 共有信息（网络拓扑、队形几何）
    route: RouteS = None                                               # 长机航线配置
    control_period_s: float = 0.05                                      # 上层注入的控制算法处理周期，单位 s
    # 各单元的 InitS（如 PID 增益）由实体在此聚合或在 init 内构造，分发给对应单元

@dataclass
class EntityInputS:                # 每拍由平台喂入的边界输入
    selfState: MotionProfS = None                              # 本体运动状态（传感器）
    inbox: list[MessageEnvelope] = field(default_factory=list) # 收到的报文
    remote: RemoteCmdS = None                                  # 遥控指令

@dataclass
class EntityOutputS:               # 每拍交给平台的边界输出
    selfAccCmd: AccInEarthS = None                             # 三轴加速度指令（送控制/执行）
    selfCmd: MotionProfS = None                                # 本机目标运动状态（位置解算结果，东北天）
    controlDiag: PosTrackDiagS = None                          # 控制诊断量（叶类型定义见《1-Context和叶类型.md》）
    outbox: list[MessageEnvelope] = field(default_factory=list)# 待发报文
```

## 2.3 边界 I/O 与配置的归属

按《0-HLD.md》3.3，以下数据不进 `Context`，由实体持有：

- **边界输入** `inbox`、`remote`：实体在 `init` 时各建一个持有对象（`remote` 用 `RemoteCmdS` 包装以满足端口绑定粒度）；收消息/任务编排单元的对应端口绑定到这些实体持有对象。
- **边界输出** `outbox`：实体持有；发消息单元的输出端口绑定到它。
- **配置** `FormCommInitS`、`FormSelfInitS`、`RouteS`、`control_period_s`、各单元 InitS：仿真控制从配置文件读取 `formation` 生成 `commInit.formPat` / `commInit.formPos`，读取 `route` 生成 `RouteS`，按 `step_s * algorithm_decimation` 计算 `control_period_s`，实体在 `init` 时持有并注入各单元（例如注入僚机 `SlotGeometry`，把 `RouteS` 注入长机 `LeaderRoute`，把 `control_period_s` 注入 `PidCompose` 的三轴 PID，把 `commInit.netWork` 与 `selfInit.id` 注入长机 `Outbound`）。

## 2.4 init 与 step 的职责

- `init`：① 实例化 `Context`；② 实例化本实体方式所需的各单元子类及其 InputS/OutputS；③ 实例化边界 I/O 持有对象；④ 调用各单元 `init` 注入配置；⑤ 把各单元 I/O 的每个端口绑定到 `Context` 子对象或边界持有对象（绑定一次，之后不再重赋值）。
- `step`：① 把 `u`（本体状态、inbox、remote）写入对应位置（`Context.selfState`、实体 inbox、实体 remote）；② 按本方式顺序依次调用各单元 `step`；③ 把结果（`Context.selfAccCmd`、`Context.selfCmd`、各单元 `diag`、实体 outbox）读出到 `y`。

数据边界约定：

```text
算法组/流程组
  -> 写各自 OutputS 的 y.xxx 或 y.diag
  -> 对象组绑定到 Context 或对象组持有的诊断对象
  -> EntityOutputS 显式带出 selfAccCmd / selfCmd / controlDiag / outbox
  -> SimCtrl 只读取 EntityOutputS，不直接读取对象组内部 Context
```

其中 `Context` 只保存控制闭环需要跨拍保留或被多个单元读写的业务状态；`controlDiag` 是对本拍控制过程的观测副本，供日志和 UI 使用，不反向参与算法计算。

# 三、各场景

## 3.1 场景一：领航跟随保持

验证场景说明：1 个航段 + 集中式通信 + 长机僚机 + PID 控制 + 三角队形。长机与僚机复用同一批算法/流程单元，但因角色不同，选用的子类与调用顺序不同，部分步骤为空或仅占位。两机的唯一耦合点是长机广播的报文经平台搬运进入僚机的 `inbox`（跨拍传递，存在至少 1 拍信息滞后，见《0-HLD.md》4.2）。

本场景有两种实体方式：长机 `LeaderEntity`、僚机 `FollowerEntity`。

### 3.1.1 各单元使用的子类

| 单元 | 长机 LeaderEntity | 僚机 FollowerEntity |
|---|---|---|
| 收消息 Inbound | 不使用（长机不收） | LeaderFollower（接收并解析长机广播） |
| 任务编排 FormationTask | Hold（恒“保持+三角”） | 不使用（模态来自长机广播） |
| 轨迹规划 TraPlan | LeaderRoute（按航线选当前航段） | Noop（占位） |
| 位置解算 PosCalc | RouteInterp（航线插值） | SlotGeometry（槽位几何） |
| 跟踪 PosTrack | PidCompose（内部求偏差 + Ctrl=PID） | PidCompose（内部求偏差 + Ctrl=PID） |
| 发消息 Outbound | LeaderBroadcast（广播） | 不使用（僚机不发） |

### 3.1.2 调用顺序（一拍 step 内）

长机 LeaderEntity：

```
任务编排(Hold) → 轨迹规划(LeaderRoute) → 位置解算(RouteInterp)
→ 跟踪(PidCompose) → 发消息(LeaderBroadcast)
```

僚机 FollowerEntity：

```
收消息(LeaderFollower) → [轨迹规划(Noop) 占位] → 位置解算(SlotGeometry)
→ 跟踪(PidCompose)
```

说明：长机自产模态、推进航线，最后广播；不收消息。僚机先收并解析长机广播，据此算槽位目标，不编排、不规划、不发消息。两机的位置解算、跟踪两步同构，只是位置解算的目标来源不同（长机来自航段，僚机来自长机状态+槽位）；跟踪内部已含求偏差。

### 3.1.3 端口绑定要点

绑定关系由“各单元选了哪个子类”即可确定（各端口绑到 `Context` 哪个字段见《2-算法组.md》《3-流程组.md》），此处只列两机的差异点：

- 长机：`Outbound` 输入端口绑 `Context.cmd`、`Context.selfState`，输出端口绑实体 `outbox`；无 `Inbound`。
- 僚机：`Inbound` 输入端口绑实体 `inbox`，输出端口绑 `Context.leaderState`、`Context.cmd`；位置解算用 `SlotGeometry`，其 `leaderState` 端口绑 `Context.leaderState`（即收消息写入的值）；无 `Outbound`、无 `FormationTask`。

### 3.1.4 场景级联试 / 验证入口

- 联试目标：长机僚机各自实例化、绑定、按上述顺序跑通一拍至多拍，数据沿 `Context` 正确流转；长机 `outbox` 经平台搬运进入僚机 `inbox` 后，僚机能据此跟随。
- 入口：分别构造 `LeaderEntity`、`FollowerEntity`，给定初始配置（机 ID、拓扑、配置文件中的队形几何、控制周期、PID 增益、航线），按拍驱动 `step`，平台在拍间搬运报文。
- 通过判据的方向：僚机目标位置稳定收敛到“长机位置 + 三角槽位偏移”，跟踪误差收敛。
- 具体仿真结果与分析见《5-用例-领航跟随保持.md》。
