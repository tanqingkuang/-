# 集结待命盘旋方案

> 目标场景：仿真开始后各机先在本地盘旋等待；收到“开始集结”命令后，各机沿本地盘旋圆到集结盘旋圆的切线进入集结。

---

## 1. 背景与目标

现有集结流程在仿真开始后直接进入 `RALLY`：各机从当前位置计算到集结盘旋圆的切入点，然后进入现有 `JOINING/CATCHUP/LOOSE/HOLD` 流程。这个方案适合自动演示，但不支持“先待命、后集结”的交互流程。

本方案新增一层集结控制语义：

```text
开始运行
→ 各机在本地盘旋圆盘旋等待

开始集结
→ 各机从本地盘旋圆切出，沿最短可行公切线进入集结盘旋圆
→ 复用现有集结同步、追赶、松散收敛、保持流程
```

本版只实现一次性开始集结，不实现运行期取消或回退。开始集结后，流程单向进入集结和队形保持；若需要重新开始，只通过现有 `reset()` 回到配置初始状态。

---

## 2. 总体原则

1. **开始运行与开始集结分离**：`start()` 只启动仿真推进，集结角色默认进入本地盘旋等待；真正进入集结由新增命令触发。
2. **命令下发给所有集结节点**：`start_rally()` 同时作用于 `rally_leader` 与 `rally_follower`，因为每架机都要独立从本地盘旋切换到切线转移和集结盘旋状态。
3. **盘旋方向完整方案固定为 CCW**：本地盘旋圆、集结盘旋圆均按逆时针飞行。
4. **盘旋半径复用现有配置**：本地盘旋圆半径直接使用 `rally_cfg.loiter_radius_m`，不新增半径配置。
5. **不支持取消/回退**：`STANDBY → ACTIVE → HOLD` 是单向流程；进入集结后不再返回本地盘旋等待。
6. **显示边界明确**：snapshot 与 GUI 只保留本地盘旋圆、集结盘旋圆；运行期切线路径由 `RallyJoinPos` 内部规划和执行，不输出静态切点或切线预览，避免辅助线与实际轨迹重复。
7. **几何异常不崩溃**：两圆圆心近零时直接进入集结圆盘旋；其他公切线无解场景退回当前点到集结圆切线，仍无解时直飞 `M_i`。

---

## 3. 状态机

### 3.1 对外状态

建议对 UI 和日志暴露以下状态：

```text
LOCAL_LOITER      本地盘旋等待
RALLY_TRANSIT     本地盘旋圆到集结盘旋圆的切线转移
RALLY_LOITER      已进入集结盘旋圆，等待同步
RALLY_EXITED      已从集结盘旋圆切出，等待全队同步
CATCHUP           追赶松散槽位
LOOSE             松散队形稳定判定
HOLD              编队保持
```

其中 `CATCHUP/LOOSE/HOLD` 复用现有 Rally 任务语义。`LOCAL_LOITER`、`RALLY_TRANSIT`、`RALLY_LOITER`、`RALLY_EXITED` 是新增的显示/诊断阶段，`RALLY_LOITER` 对应现有 `RallyJoinPos.state == LOITERING`。

### 3.2 状态转换

```text
load_config
  → READY

start()
  → LOCAL_LOITER

start_rally()
  LOCAL_LOITER → RALLY_TRANSIT
  RALLY_TRANSIT / RALLY_LOITER / RALLY_EXITED / CATCHUP / LOOSE → 提示“已在集结中”
  HOLD → 返回 ERR_INVALID_STATE，不改变当前阶段；GUI 按钮禁用

进入集结圆
  RALLY_TRANSIT → RALLY_LOITER

个体切出集结圆
  RALLY_LOITER → RALLY_EXITED

全队切出同步完成
  RALLY_EXITED → CATCHUP → LOOSE → HOLD

reset()
  → 回到配置初始状态，重新等待 start()
```

内部实现映射见 §3.3。

### 3.3 内部状态表示

`LOCAL_LOITER` / `RALLY_TRANSIT` 等字符串只用于 UI、日志和 `NodeState.rally_phase`，不直接作为编队主阶段下发。底层待命阶段使用 `FormStageE.STANDBY` 表达，因此 `FormStageE`（`src/algorithm/context/leaf_types.py`）当前包含 `NONE/RALLY/HOLD/DISBAND/STANDBY` 五值，其中：

- `RemoteCmdS.stage`（`remote`）是 controller 直接写入每个 `_NodeAlgorithm` 的本地遥控指令，不经过 `comm.py` 仿真链路；
- `cxt.cmd.stage` 是长机计算后经 `FormationOutbound` 广播、由僚机 `FormationInbound` 解析的编队阶段，是对外协议的一部分。

`DISBAND=3` 只为未来解散能力预留连续编号，当前不在实体合法状态序列和策略表中，不能下发执行。实体边界和 Rally 任务会在修改本拍状态前明确拒绝该值，禁止它落入默认集结路径。`STANDBY=4` 保留既有线协议值，专门表示集结节点已经开始本地待命盘旋，但尚未进入 `RALLY` 集结执行。故障“重构”仅由快照层根据节点健康状态推导，不再占用编队主阶段枚举。

同时新增一个只在 controller / `_NodeAlgorithm` 内部使用的运行模式：

```text
RallyRunModeE
  STANDBY   本地盘旋等待，对应对外显示态 LOCAL_LOITER
  ACTIVE    已收到 start_rally()，RallyJoinPos 在 FLYING 内部执行公切线转移
```

`rally_run_mode == STANDBY` 时，`_NodeAlgorithm.step()` 向实体传入有效 `remote.stage = FormStageE.STANDBY`。实体仍然走原来的主体流程槽位，但位置解算策略统一为 `RallyJoinPos` 的内部 `STANDBY` 状态，不新增独立流程或第二个待命位置解算对象；`Rally` 任务槽位只输出 `cmd.stage = STANDBY`，不推进 JOINING/CATCHUP/LOOSE 子流程。长机和僚机仍按原通信链路正常解析与广播，广播阶段也明确携带 `STANDBY`。`start_rally()` 把目标节点的 `rally_run_mode` 和 `remote.stage` 一起切到 `ACTIVE`/`RALLY`。当前版本不提供 `ACTIVE → STANDBY` 的反向切换。

`LOCAL_LOITER/RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED` 只属于显示与诊断层，不是 `FormStageE`、`RemoteCmdS.stage` 或广播协议的新枚举值。快照层统一通过每节点 `NodeState.rally_phase` 暴露集结细分阶段，其中 `FormStageE.STANDBY` 对外映射为 `LOCAL_LOITER`。下表是 `_NodeAlgorithm.current_rally_phase_str()` 的唯一规范定义，按顺序自上而下匹配：

```text
非集结节点                                                                         → ""
controller.run_state == READY                                                      → ""
rally_run_mode == STANDBY                                                          → "LOCAL_LOITER"
cmd.stage == HOLD                                                                  → "HOLD"
cmd.stage == RALLY 且 cmd.step == CATCHUP/LOOSE                                    → "CATCHUP" / "LOOSE"
cmd.stage == RALLY 且 cmd.step == JOINING 且 RallyJoinPos.state == FLYING              → "RALLY_TRANSIT"
cmd.stage == RALLY 且 cmd.step == JOINING 且 RallyJoinPos.state == LOITERING        → "RALLY_LOITER"
cmd.stage == RALLY 且 cmd.step == JOINING 且 RallyJoinPos.state == EXITED           → "RALLY_EXITED"
```

全局 `control_report` 仍保持现有四类中文大状态（`待命/集结/保持/重构`），不新增 `LOCAL_LOITER` 等细分值。推导规则需要从 `NodeState.rally_phase` 或 `_NodeAlgorithm.rally_run_mode` 获取集结待命语义，不能只看 `cxt.cmd.stage`：

```text
任一节点健康异常                         → control_report = "重构"
任一集结节点 rally_phase 为 RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE → "集结"
任一节点 stage == HOLD 或 rally_phase == HOLD                                 → "保持"
存在集结节点且 rally_phase == LOCAL_LOITER                                    → "待命"
无算法节点                                                               → "待命"
其余已加载但未运行或未命中上述规则的场景                                  → "待命"
```

`READY` 阶段不表示算法已进入本地盘旋，因此 `NodeState.rally_phase` 保持空字符串，GUI 不按集结阶段绘制辅助几何；只有 `start()` 之后，集结节点才从 `STANDBY` 映射为 `LOCAL_LOITER` 并显示本地待命/集结辅助几何。这条约束用于避免加载配置后用户误以为飞机已经开始盘旋，也避免 `LOCAL_LOITER` 时 `cxt.cmd.stage == NONE` 被现有默认分支误显示为“保持”。混合场景中保持优先于待命：如果同一配置里普通 `leader/wingman` 已经处于 `HOLD`，而集结节点仍在 `LOCAL_LOITER`，全局 `control_report` 显示“保持”，集结节点自己的 `NodeState.rally_phase` 仍显示 `LOCAL_LOITER`。这是保留现有四类全局回报语义的设计取舍，细分状态以节点级字段为准。

`RallyJoinPos` 的公开状态仍只有 `STANDBY/FLYING/LOITERING/EXITED`。开始集结后，`FLYING` 内部依次执行 `ARC_TO_TANGENT` 和 `LINE_TO_RALLY_ENTRY` 两个私有子阶段；两者对任务编排、通信和 GUI 均继续表现为 `FLYING/RALLY_TRANSIT`，不新增流程对象或公开枚举。转场判据见 §6.5。

## 4. 控制命令

### 4.1 `start()`

现有 `SimulationController.start()` 保持“启动仿真线程/推进时间”的职责，但对集结角色的初始远控阶段需要调整：

```text
当前：rally_leader / rally_follower 初始 remote.stage = RALLY
目标：rally_leader / rally_follower 初始 rally_run_mode = STANDBY，持久 remote.stage = NONE；实际 step 时由 `_NodeAlgorithm` 派生有效 remote.stage = STANDBY
```

普通 `leader/wingman` 保持现有 `HOLD` 语义不变。

### 4.2 `start_rally()`

新增控制器命令，负责把所有集结节点从 `LOCAL_LOITER` 切到 `RALLY_TRANSIT`。

语义：

- 未加载配置：返回 `ERR_NO_CONFIG`
- 已结束：返回 `ERR_INVALID_STATE`
- 非集结配置：返回 `ERR_INVALID_STATE`
- 已在集结中：返回 `ERR_INVALID_STATE`，消息为“已在集结中”
- 已进入 `HOLD`：返回 `ERR_INVALID_STATE`
- 可开始时：返回 `OK`，下一算法拍生效

仿真运行状态对 `start_rally()` 的影响：

- `READY`（已加载配置但未 `start()`）：返回 `ERR_INVALID_STATE`，提示“请先开始运行”；集结节点此时尚未进入 `LOCAL_LOITER`，没有可切换的运行态
- `PAUSED`：允许调用，只切 `rally_run_mode`/`remote.stage`，不推进仿真时间；恢复运行后从下一算法拍开始按新状态推进
- `RUNNING`：下一算法拍生效（同上）

如果 `start_rally()` 在首个仿真 tick 前触发（包括启动后立即点击，或 `run_until_complete()` 批处理自动开始集结），控制器先执行一次 STANDBY 算法预热，只让各实体按当前状态建立待命圆和待命控制缓存，不推进通信时钟、模型或仿真时间；随后再切 `ACTIVE/RALLY`。这样批处理与 GUI 正常等待后的路径合同一致，都从已锁存的待命圆规划公切线。已经推进过 tick 或重复点击时不执行预热。

`READY` 快照可以保留按初始位置/航向预计算的本地盘旋圆数据，供调试、测试或后续预览功能复用；但当前正式 GUI 不在 `READY` 阶段绘制该几何，避免把“已加载配置”误表达成“已进入本地盘旋”。只有 `start()` 后实体才真正输出本地盘旋指令，GUI 才按 `LOCAL_LOITER` 等节点阶段绘制辅助几何。

## 5. 本地盘旋圆

### 5.1 生成规则

本地盘旋圆不把飞机当前位置当圆心，而是把当前位置当作圆上的当前点。圆心由当前航向、盘旋方向和半径反推。

完整方案固定 CCW：

```text
当前位置 P = (east, north)
当前水平航向 ψ
盘旋半径 R = rally_cfg.loiter_radius_m

CCW 圆心 C_local = P + left_normal(ψ) × R
```

这样开始运行时，飞机可以从当前航向自然接入圆周，不需要先飞到圆外或发生明显掉头。

### 5.2 本地盘旋指令

本地盘旋阶段输出圆周上的目标点、切向速度和向心角速度，速度限制沿用现有 `RallyJoinPos` 的盘旋速度上下限：

```text
loiter_speed_min_mps / loiter_speed_max_mps
```

初始盘旋速度以当前速度为参考，并夹到盘旋速度上下限内。

## 6. 开始集结的切线规划

### 6.1 目标

开始集结时，每架飞机从自己的本地盘旋圆切出，沿一条直线切线进入自己的集结盘旋圆。

```text
本地盘旋圆 C_local
→ 切出点 T_local
→ 直线切线
→ 集结盘旋圆切入点 T_rally
→ 集结盘旋圆上的目标槽位 M_i
```

### 6.2 定向公切线

两个圆均按 CCW、半径均取 `loiter_radius_m`，因此复用 `arc_path.common_tangent()` 求同向外公切线。该函数从两条几何外公切线中选择“从 `T_local` 指向 `T_rally`，且与本地圆 CCW 前进切向同向”的唯一可飞切线；另一条的切向方向背离集结圆，不能作为本次转移路径。

### 6.3 一次性规划与 ETA

每架飞机在 `STANDBY → FLYING` 的第一拍独立规划一次，并锁存 `T_local`、`T_rally`、两点对应圆周角和公切线长度；后续不按实时位置重算，避免目标漂移。完整 ETA 按真实飞行路径估算：

```text
总路径长度 =
  当前点沿本地圆到 T_local 的 CCW 弧长
  + T_local 到 T_rally 的直线段长度
  + T_rally 沿集结圆到 M_i 的 CCW 弧长
```

待命圆弧使用锁存待命速度，直线段与现有 FLYING 共用 `slow_radius_m` 近场降速和最小速度托底，集结圆弧使用 `approach_speed_mps` 粗估。进入直线段后，ETA 改为“当前位置到 `T_rally` 的剩余距离 + `T_rally` 到 `M_i` 的圆弧”，并逐拍修正。

### 6.4 无解兜底

待命圆与集结圆圆心距离不大于 `_EPSILON_HORIZ`（当前为 `0.5m`）时，两圆可视为重合，飞机已经位于集结圆上，直接进入 `LOITERING`，不生成无意义的直飞段。

其他情况下，如果 `common_tangent()` 因几何或数值退化返回无解，则退回现有点到圆逻辑：

```text
当前位置 → 集结盘旋圆外点切入点
```

该切入点同样只计算并锁存一次。若当前点位于集结圆内或圆上，点到圆切线也无解，则继续退化为直飞 `M_i`。这些兜底保证开始集结命令不会因几何退化导致算法中断。

### 6.5 转场判据与交接

`RallyJoinPos.FLYING` 内部按以下判据递进：

1. `ARC_TO_TANGENT`：继续输出待命圆 CCW 指令；待命圆当前位置到 `T_local` 的剩余 CCW 角不大于命名常量 `10°` 时，同一拍切换到直线段。`10°` 用于限制提前切线造成的航向差；它与集结圆入口的 `5°` 捕获约束分别生效。实现同时比较上一拍剩余角，若离散步进跨过零点，也立即切换，避免漏过窗口后多绕一圈。
2. `LINE_TO_RALLY_ENTRY`：持续直飞锁存的 `T_rally`；满足现有 `d_3d < min(arrival_radius_m, _arc_capture_radius_m)` 后进入 `LOITERING`。`_arc_capture_radius_m` 仍由 `loiter_radius_m` 和 `5°` 航向跳变上限反解。

两个私有子阶段对外始终报告 `FLYING/RALLY_TRANSIT`；只有进入集结圆 `LOITERING` 后，显示态才切换为 `RALLY_LOITER`。

## 7. 高度分层与收敛

### 7.1 目标

集结飞行过程中，多机可能从不同方向同时进入集结圆。为了降低近距离交汇风险，在形成松散队形之前，各机应保持高度错开；形成松散队形后，再逐步收敛到目标队形高度。

### 7.2 高度策略

完整方案采用以下阶段策略：

```text
LOCAL_LOITER / RALLY_TRANSIT / RALLY_LOITER / RALLY_EXITED / CATCHUP
  使用高度分层

LOOSE
  开始向目标队形高度收敛

LOOSE / HOLD
  使用正常队形高度
```

**实现约束**：`CATCHUP` 阶段复用与 `LOOSE/HOLD` 相同的 `SlotGeometry` 算法。`SlotGeometry` 的正常目标使用三维 FUR（前、上、右）槽位，`slot.y` 是上法向间隔，并非固定世界高度差；爬升/下降时目标 ENU 高度还会包含 `slot.x/y` 随倾角旋转后的分量。`CATCHUP` 使用可选的 ENU 高度覆盖，进入 `LOOSE` 后恢复正常队形高度。

### 7.3 高度分层配置

新增可选配置项：

```json
"rally_cfg": {
  "altitude_separation_m": 60.0
}
```

未配置时默认 `60.0m`。

高度分层建议围绕长机高度上下交错，而不是单调递增，避免多机数量增加时整体高度偏移过大。例如基于当前五机队形文件槽位顺序（A01、A02、A03、A04、A05，且 A03 为 `rally_leader`）：

```text
A03 leader: 基准高度
A01: 基准高度 + 60m
A02: 基准高度 - 60m
A04: 基准高度 + 120m
A05: 基准高度 - 120m
```

实际偏置顺序按队形槽位列表稳定生成：先找到 `rally_leader` 对应槽位置为 0，其余节点按槽位文件中的顺序（去掉 leader）交错分配 `+1, -1, +2, -2 ...` 个高度间隔。这样同一配置每次加载得到相同分层，也避免节点数组顺序变化影响高度分配。

**基准高度**取当前有效 `route[0]`（集结区起点 A）的高度，不取长机实时高度或长机初始配置高度：分层需要在开始集结前稳定确定，用长机实时高度会随长机爬升/下降而变化；采用避障规划结果后，覆盖航线会成为当前有效航线并重新初始化该基准。`route[0].pos.h` 与航向、切线复用同一基准点，各机在真正起飞前就能拿到，天然一致。

### 7.4 高度收敛

进入 `LOOSE` 后，目标高度切回正常队形高度：

```text
formation_altitude = leader.h + slot.y
```

完整方案可先依赖现有 `max_climb_rate_mps` / `max_descent_rate_mps` 做物理限幅，避免瞬时高度跳变。如果后续观察到高度命令突变过大，再增加高度融合因子：

```text
alt_target = mix(layer_altitude, formation_altitude, progress)
```

## 8. 界面与显示

### 8.1 新增按钮

在正式 PySide6 GUI 中新增一个控制按钮：

```text
开始集结
```

按钮状态建议：

- 未加载配置：禁用；
- 非集结配置：禁用；
- `LOCAL_LOITER`：启用“开始集结”；
- 集结中（`RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE`）：“开始集结”保持可点击，点击返回“已在集结中”提示，不禁用；
- `HOLD`：禁用“开始集结”。

现有“开始/暂停/单步/重置”保留仿真运行控制语义。

### 8.2 俯视图显示

当前版本显示以下元素：

- 本地盘旋圆；
- 集结盘旋圆；
- 当前集结状态。

显示数据由 snapshot 提供，GUI 不做几何计算。沿用 `SimulationSnapshot.rally_geometry` 字段名，值类型为 `RallyPlanGeometryState`。当前正式字段契约如下：

```text
node_id
local_center_east_m / local_center_north_m / local_radius_m       本地盘旋圆
rally_center_east_m / rally_center_north_m / rally_radius_m       集结盘旋圆
```

`SimulationController` 在初始化时按当前生效航线和初始状态生成两个圆，运行中复用该快照几何。实际切点、松散目标点和几何退化状态属于算法内部运行数据，不进入 GUI 契约。当前集结状态统一读取 `NodeState.rally_phase`，避免在节点状态与几何结构中重复维护。

## 9. 模块影响

### 9.1 runner / controller

需要新增控制命令：

```text
SimulationController.start_rally()
```

`_NodeAlgorithm` 需要新增本地运行模式字段（`RallyRunModeE`，见 §3.3），并配合新增的 `FormStageE.STANDBY` 使用：

```text
rally_run_mode: STANDBY | ACTIVE
remote.stage: NONE/STANDBY/RALLY/HOLD
```

`rally_run_mode == STANDBY` 时，runner 向实体传入有效 `remote.stage == STANDBY`；`rally_run_mode == ACTIVE` 时，`remote.stage == RALLY` 内部继续由现有 Rally 任务推进 `JOINING/CATCHUP/LOOSE`，不新增 `cmd.step` 值。

`_NodeAlgorithm.current_rally_phase_str()` 需要从 `rally_run_mode`、当前驱动单元、`RallyJoinPos.state` 与 `cmd.step` 派生 §3.3 的规范字符串，并写入 `NodeState.rally_phase`。该方法取代现有 `"JN·{FLY/LOIT/EXIT}"` 拼接逻辑，旧标签不再保留，避免 `_NodeAlgorithm` 内同时存在两套阶段字符串语义。`SimulationController._make_snapshot_unlocked()` 在 `run_state == READY` 时必须压制集结节点的 `rally_phase` 输出为空字符串；正式 GUI 不在 READY 阶段绘制集结辅助几何。`SimulationController._derive_control_report_unlocked()` 不能再只依赖 `current_stage()`，需要按 §3.3 的快照契约识别 `LOCAL_LOITER` 为“待命”、识别 `RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE` 为“集结”。

`SimulationController` 在初始化模块时根据当前生效航线和初始状态预计算 `SimulationSnapshot.rally_geometry`，运行中继续复用这份圆几何。采用避障覆盖航线会重建模块，因此两个圆同步按覆盖航线更新。切点和退化状态不进入 snapshot，也不新增运行期几何刷新接口；真实转场始终由算法层按实时状态执行。

`SimulationController._prime_rally_standby_unlocked()` 只处理首个 tick 前的入口一致性：节点仍处于 `LOCAL_LOITER` 运行模式时预热一次 `_run_formation_algorithms_unlocked()`，随后由原有 `start_rally()` 切换远控阶段。预热不替代正常 tick，也不引入新的 Entity 主流程。

### 9.2 algorithm

位置解算槽位不新增流程对象，公切线规划和转移统一扩展在 `RallyJoinPos` 内：

```text
RallyJoinPos: STANDBY / FLYING(ARC_TO_TANGENT / LINE_TO_RALLY_ENTRY) / LOITERING / EXITED
```

边界约定：

- `RallyJoinPos.STANDBY`：负责当前位置本地待命盘旋，圆心由进入待命那一拍的当前位置与航向反推；它与 `ARC_TO_TANGENT`、`LOITERING` 复用同一套“圆上投影目标 + 切向速度 + 曲率前馈 + 垂向速度”的几何口径；
- `RallyJoinPos.FLYING`：开始集结时一次性规划并锁存定向公切线，内部完成待命圆切出和集结圆切入，不增加公开状态或外部切点注入接口；
- `RallyJoinPos.LOITERING/EXITED`：继续负责集结圆盘旋同步和从 `M_i` 沿任务航向切出；
- `SlotGeometry`：新增可选高度覆盖输入端口，供 `CATCHUP` 阶段写入分层高度（见 §7.2）。

### 9.3 通信行为

`LOCAL_LOITER` 期间僚机继续由 `FormationOutbound` 发送 `FOLLOWER_STATUS`（不新增静默分支，保持每帧广播的现有实现方式不变）。`RallyJoinPos.STANDBY` 必须写入 `cxt.selfCmd`，使待命期间上报的 `pos_err_m` 表示到本地盘旋投影目标的误差，不能沿用旧集结目标。待命期间报文中的 `rally_state` 必须显式写为 `RALLY_STATE_STANDBY`（字符串值 `STANDBY`），避免 UI/日志把待命误显示为正在飞向集结。长机在自身 `rally_run_mode == STANDBY` 时仍正常解析僚机回报，并继续通过 `FormationOutbound` 广播 `cmd.stage = STANDBY`；任务槽位只保持 `STANDBY`，因此不会推进到“已在集结”的子流程。

实现时在 `rally_join_pos.py` 与相关测试中新增常量：

```python
RALLY_STATE_STANDBY = "STANDBY"
```

并同步更新 `FormationOutbound` 私有输入端口及 `FollowerStateS.rally_state` 的注释，取值范围固定为 `STANDBY / FLYING / LOITERING / EXITED`。业务代码不得直接散写裸字符串 `"STANDBY"`。

### 9.4 GUI

GUI 只新增按钮和圆几何绘制，不直接参与几何规划。控制入口通过 `ControllerSimulationAdapter` 转发到 controller。GUI 侧契约如下：

- `ControllerSimulationAdapter._convert_snapshot()`：只转换本地盘旋圆和集结盘旋圆字段；
- `src/ui/gui/view_models.py` 的 `RallyGeometryView`：只保留 `local_*` 与集结圆 `center_x/center_y/radius` 字段；
- `src/ui/gui/top_view.py`：绘制过滤规则改为 `rally_phase in {"LOCAL_LOITER", "RALLY_TRANSIT", "RALLY_LOITER", "RALLY_EXITED", "CATCHUP"}`，不再使用旧的 `rally_phase.startswith("JN")`；`HOLD` 后不绘制集结几何；
- `LOCAL_LOITER` 显示本地盘旋圆；`RALLY_TRANSIT` 显示本地圆和集结圆；`RALLY_LOITER/RALLY_EXITED/CATCHUP` 保留集结圆。静态切线、切入点、切出点和槽位标记均不绘制，实际飞行过程通过轨迹显示。

## 10. 测试建议

### 10.1 单元测试

几何测试：

- 当前点 + 航向 + 半径能生成正确的 CCW 本地圆；
- 两圆有可行公切线时，能选出与 CCW 前进方向一致的定向外公切线，并锁存两端切点；
- 两圆圆心近零时直接进入 `LOITERING`；公切线无解时退回锁存的当前点到集结圆切线；
- 切线两端与对应圆相切；
- CCW 弧长计算方向正确；
- 进入 `10°` 切出窗口或单拍跨过切点时切换到直线段，不多绕一圈；
- `rally_geometry` 快照字段只包含本地圆和集结圆，并确认不存在静态切点、槽位点和退化标志字段。

高度测试：

- `altitude_separation_m` 缺省为 `60.0m`；
- 多机高度分层稳定且围绕长机上下错开；
- `LOCAL_LOITER/RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP` 使用分层高度；
- `LOOSE/HOLD` 回到队形高度。

状态机测试：

- `start()` 后集结节点进入 `LOCAL_LOITER`；
- `start_rally()` 从 `LOCAL_LOITER` 进入 `RALLY_TRANSIT`；
- 集结中重复 `start_rally()` 返回“已在集结中”；
- `HOLD` 后 `start_rally()` 返回 `ERR_INVALID_STATE`，不改变当前阶段；
- `LOCAL_LOITER` 时 `control_report == "待命"`，`NodeState.rally_phase == "LOCAL_LOITER"`；
- `RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE` 时 `control_report == "集结"`；
- `reset()` 后回到初始待运行状态。

### 10.2 GUI 测试

- 加载集结配置后显示“开始集结”按钮；
- 点“开始运行”后俯视图显示各机本地盘旋圆；
- 点“开始集结”后俯视图显示本地圆、集结圆和实际飞行轨迹，不显示静态切线或点标记；
- `HOLD` 后“开始集结”按钮禁用。

## 11. 分阶段落地建议

以下阶段是内部开发拆分；完整验收必须覆盖本方案的控制流程、切线规划、圆几何显示和高度分层。单独完成第一阶段只表示基础控制链路可运行，不代表本功能已经完成。

第一阶段：

- 新增控制命令和状态机；
- 实现本地盘旋；
- GUI 增加按钮和基础状态提示。

第二阶段：

- 实现本地圆到集结圆的切线规划；
- 输出并绘制本地圆、集结圆，切线只作为算法内部运行路径；
- 加入几何退化兜底。

第三阶段：

- 加入高度分层；
- 调整 `LOOSE` 之后的高度收敛；
- 补充多机避撞相关回归测试。

该拆分可以降低单次改动风险，但最终行为仍按本文档的完整方案验收。
