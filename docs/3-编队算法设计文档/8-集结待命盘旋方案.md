# 集结待命盘旋方案

> 目标场景：仿真开始后各机先在本地盘旋等待；收到“开始集结”命令后，各机沿本地盘旋圆到集结盘旋圆的切线进入集结。

---

## 1. 背景与目标

现有集结流程在仿真开始后直接进入 `RALLY`：各机从当前位置计算到集结盘旋圆的切入点，然后进入现有 `JOINING/CATCHUP/LOOSE/COMPRESS/HOLD` 流程。这个方案适合自动演示，但不支持“先待命、后集结”的交互流程。

本方案新增一层集结控制语义：

```text
开始运行
→ 各机在本地盘旋圆盘旋等待

开始集结
→ 各机从本地盘旋圆切出，沿最短可行公切线进入集结盘旋圆
→ 复用现有集结同步、追赶、压缩、保持流程
```

本版只实现一次性开始集结，不实现运行期取消或回退。开始集结后，流程单向进入集结和队形保持；若需要重新开始，只通过现有 `reset()` 回到配置初始状态。

---

## 2. 总体原则

1. **开始运行与开始集结分离**：`start()` 只启动仿真推进，集结角色默认进入本地盘旋等待；真正进入集结由新增命令触发。
2. **命令下发给所有集结节点**：`start_rally()` 同时作用于 `rally_leader` 与 `rally_follower`，因为每架机都要独立从本地盘旋切换到切线转移和集结盘旋状态。
3. **盘旋方向完整方案固定为 CCW**：本地盘旋圆、集结盘旋圆均按逆时针飞行。
4. **盘旋半径复用现有配置**：本地盘旋圆半径直接使用 `rally_cfg.loiter_radius_m`，不新增半径配置。
5. **不支持取消/回退**：`STANDBY → ACTIVE → HOLD` 是单向流程；进入集结后不再返回本地盘旋等待。
6. **显示与算法同源**：本地盘旋圆、集结盘旋圆、切线路径由 runner/算法层输出到 snapshot，GUI 只负责绘制，避免显示和实际飞行分叉。
7. **几何异常不崩溃**：两圆太近、重合或公切线无解时，退回旧逻辑：从当前位置按外点切线进入集结盘旋圆。

## 3. 状态机

### 3.1 对外状态

建议对 UI 和日志暴露以下状态：

```text
LOCAL_LOITER      本地盘旋等待
RALLY_TRANSIT     本地盘旋圆到集结盘旋圆的切线转移
RALLY_LOITER      已进入集结盘旋圆，等待同步
RALLY_EXITED      已从集结盘旋圆切出，等待全队同步
CATCHUP           追赶松散槽位
LOOSE             松散队形稳定判定
COMPRESS          压缩到目标队形
HOLD              编队保持
```

其中 `CATCHUP/LOOSE/COMPRESS/HOLD` 复用现有 Rally 任务语义。`LOCAL_LOITER`、`RALLY_TRANSIT`、`RALLY_LOITER`、`RALLY_EXITED` 是新增的显示/诊断阶段，`RALLY_LOITER` 对应现有 `RallyJoinPos.state == LOITERING`。

### 3.2 状态转换

```text
load_config
  → READY

start()
  → LOCAL_LOITER

start_rally()
  LOCAL_LOITER → RALLY_TRANSIT
  RALLY_TRANSIT / RALLY_LOITER / RALLY_EXITED / CATCHUP / LOOSE / COMPRESS → 提示“已在集结中”
  HOLD → 返回 ERR_INVALID_STATE，不改变当前阶段；GUI 按钮禁用

进入集结圆
  RALLY_TRANSIT → RALLY_LOITER

个体切出集结圆
  RALLY_LOITER → RALLY_EXITED

全队切出同步完成
  RALLY_EXITED → CATCHUP → LOOSE → COMPRESS → HOLD

reset()
  → 回到配置初始状态，重新等待 start()
```

内部实现映射见 §3.3。

### 3.3 内部状态表示

`LOCAL_LOITER` / `RALLY_TRANSIT` 等字符串只用于 UI、日志和 `NodeState.rally_phase`，不直接作为编队主阶段下发。底层待命阶段用新增的 `FormStageE.STANDBY` 表达，因此 `FormStageE`（`src/algorithm/context/leaf_types.py`）当前包含 `NONE/RALLY/HOLD/RECONFIG/STANDBY` 五值，其中：

- `RemoteCmdS.stage`（`remote`）是 controller 直接写入每个 `_NodeAlgorithm` 的本地遥控指令，不经过 `comm.py` 仿真链路；
- `cxt.cmd.stage` 是长机计算后经 `RallyLeaderBroadcast`/`RallyLeaderFollower` 广播给僚机的编队阶段，是对外协议的一部分。

`RECONFIG` 已被 `sim_control_snapshot.py` 占用为故障重构语义，不能挪用；本方案新增 `STANDBY`，专门表示集结节点已经开始本地待命盘旋，但尚未进入 `RALLY` 集结执行。

同时新增一个只在 controller / `_NodeAlgorithm` 内部使用的运行模式：

```text
RallyRunModeE
  STANDBY   本地盘旋等待，对应对外显示态 LOCAL_LOITER
  ACTIVE    已收到 start_rally()，位置解算交给切线转移 + 现有 RallyJoinPos/Rally 流程
```

`rally_run_mode == STANDBY` 时，`_NodeAlgorithm.step()` 向实体传入有效 `remote.stage = FormStageE.STANDBY`。实体仍然走原来的主体流程槽位，但位置解算策略统一为 `RallyJoinPos` 的内部 `STANDBY` 状态，不新增独立流程或第二个待命位置解算对象；`Rally` 任务槽位只输出 `cmd.stage = STANDBY`，不推进 JOINING/CATCHUP/LOOSE/COMPRESS 子流程。长机和僚机仍按原通信链路正常解析与广播，广播阶段也明确携带 `STANDBY`。`start_rally()` 把目标节点的 `rally_run_mode` 和 `remote.stage` 一起切到 `ACTIVE`/`RALLY`。当前版本不提供 `ACTIVE → STANDBY` 的反向切换。

`LOCAL_LOITER/RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED` 只属于显示与诊断层，不是 `FormStageE`、`RemoteCmdS.stage` 或广播协议的新枚举值。快照层统一通过每节点 `NodeState.rally_phase` 暴露集结细分阶段，其中 `FormStageE.STANDBY` 对外映射为 `LOCAL_LOITER`。下表是 `_NodeAlgorithm.current_rally_phase_str()` 的唯一规范定义，按顺序自上而下匹配：

```text
非集结节点                                                                         → ""
controller.run_state == READY                                                      → ""
rally_run_mode == STANDBY                                                          → "LOCAL_LOITER"
cmd.stage == HOLD                                                                  → "HOLD"
cmd.stage == RALLY 且 cmd.step == CATCHUP/LOOSE/COMPRESS                           → "CATCHUP" / "LOOSE" / "COMPRESS"
cmd.stage == RALLY 且 cmd.step == JOINING 且（切线转移单元仍在驱动输出 或 RallyJoinPos.state == FLYING） → "RALLY_TRANSIT"
cmd.stage == RALLY 且 cmd.step == JOINING 且 RallyJoinPos.state == LOITERING        → "RALLY_LOITER"
cmd.stage == RALLY 且 cmd.step == JOINING 且 RallyJoinPos.state == EXITED           → "RALLY_EXITED"
```

全局 `control_report` 仍保持现有四类中文大状态（`待命/集结/保持/重构`），不新增 `LOCAL_LOITER` 等细分值。推导规则需要从 `NodeState.rally_phase` 或 `_NodeAlgorithm.rally_run_mode` 获取集结待命语义，不能只看 `cxt.cmd.stage`：

```text
任一节点健康异常                         → control_report = "重构"
任一集结节点 rally_phase 为 RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE/COMPRESS → "集结"
任一节点 stage == HOLD 或 rally_phase == HOLD                                 → "保持"
存在集结节点且 rally_phase == LOCAL_LOITER                                    → "待命"
无算法节点                                                               → "待命"
其余已加载但未运行或未命中上述规则的场景                                  → "待命"
```

`READY` 阶段不表示算法已进入本地盘旋，因此 `NodeState.rally_phase` 保持空字符串，GUI 不按集结阶段绘制辅助几何；只有 `start()` 之后，集结节点才从 `STANDBY` 映射为 `LOCAL_LOITER` 并显示本地待命/集结辅助几何。这条约束用于避免加载配置后用户误以为飞机已经开始盘旋，也避免 `LOCAL_LOITER` 时 `cxt.cmd.stage == NONE` 被现有默认分支误显示为“保持”。混合场景中保持优先于待命：如果同一配置里普通 `leader/wingman` 已经处于 `HOLD`，而集结节点仍在 `LOCAL_LOITER`，全局 `control_report` 显示“保持”，集结节点自己的 `NodeState.rally_phase` 仍显示 `LOCAL_LOITER`。这是保留现有四类全局回报语义的设计取舍，细分状态以节点级字段为准。

切线转移单元（§6 的 `RallyTransit`）在算出集结圆切入点 `T_rally` 后，通过 `RallyJoinPos` 的公开入口注入外部切入点，`RallyJoinPos` 从 `FLYING` 起步（此时距切入点很近，通常同一拍或下一拍即满足捕获半径条件转入 `LOITERING`），不重新从当前位置计算切线——避免二次转场。转场判据细节见 §6.5。

## 4. 控制命令

### 4.1 `start()`

现有 `SimulationController.start()` 保持“启动仿真线程/推进时间”的职责，但对集结角色的初始远控阶段需要调整：

```text
当前：rally_leader / rally_follower 初始 remote.stage = RALLY
目标：rally_leader / rally_follower 初始 rally_run_mode = STANDBY，持久 remote.stage = NONE；实际 step 时由 `_NodeAlgorithm` 派生有效 remote.stage = STANDBY
```

普通 `leader/wingman` 保持现有 `HOLD` 语义不变。

### 4.2 `start_rally()`

新增控制器命令，负责把所有集结节点从 `LOCAL_LOITER` 切到 `RALLY_TRANSIT`。

语义：

- 未加载配置：返回 `ERR_NO_CONFIG`
- 已结束：返回 `ERR_INVALID_STATE`
- 非集结配置：返回 `ERR_INVALID_STATE`
- 已在集结中：返回 `ERR_INVALID_STATE`，消息为“已在集结中”
- 已进入 `HOLD`：返回 `ERR_INVALID_STATE`
- 可开始时：返回 `OK`，下一算法拍生效

仿真运行状态对 `start_rally()` 的影响：

- `READY`（已加载配置但未 `start()`）：返回 `ERR_INVALID_STATE`，提示“请先开始运行”；集结节点此时尚未进入 `LOCAL_LOITER`，没有可切换的运行态
- `PAUSED`：允许调用，只切 `rally_run_mode`/`remote.stage`，不推进仿真时间；恢复运行后从下一算法拍开始按新状态推进
- `RUNNING`：下一算法拍生效（同上）

`READY` 快照可以保留按初始位置/航向预计算的本地盘旋圆数据，供调试、测试或后续预览功能复用；但当前正式 GUI 不在 `READY` 阶段绘制该几何，避免把“已加载配置”误表达成“已进入本地盘旋”。只有 `start()` 后实体才真正输出本地盘旋指令，GUI 才按 `LOCAL_LOITER` 等节点阶段绘制辅助几何。

## 5. 本地盘旋圆

### 5.1 生成规则

本地盘旋圆不把飞机当前位置当圆心，而是把当前位置当作圆上的当前点。圆心由当前航向、盘旋方向和半径反推。

完整方案固定 CCW：

```text
当前位置 P = (east, north)
当前水平航向 ψ
盘旋半径 R = rally_cfg.loiter_radius_m

CCW 圆心 C_local = P + left_normal(ψ) × R
```

这样开始运行时，飞机可以从当前航向自然接入圆周，不需要先飞到圆外或发生明显掉头。

### 5.2 本地盘旋指令

本地盘旋阶段输出圆周上的目标点、切向速度和向心角速度，速度限制沿用现有 `RallyJoinPos` 的盘旋速度上下限：

```text
loiter_speed_min_mps / loiter_speed_max_mps
```

初始盘旋速度以当前速度为参考，并夹到盘旋速度上下限内。

## 6. 开始集结的切线规划

### 6.1 目标

开始集结时，每架飞机从自己的本地盘旋圆切出，沿一条直线切线进入自己的集结盘旋圆。

```text
本地盘旋圆 C_local
→ 切出点 T_local
→ 直线切线
→ 集结盘旋圆切入点 T_rally
→ 集结盘旋圆上的目标槽位 M_i
```

### 6.2 候选切线

两个圆之间理论上存在多条公切线：

- 同向绕飞对应外公切线；
- 反向绕飞对应内公切线；
- 每类有左右两个候选。

本方案中本地盘旋圆和集结盘旋圆均固定 CCW，因此优先选择同向外公切线。实现时仍建议枚举所有可行候选并按代价选最优，便于后续扩展 CW 或混合方向。

### 6.3 最短路径代价

“距离最近”按真实飞行路径长度计算，而不是只看两切点之间直线距离：

```text
总路径长度 =
  当前点沿本地圆到 T_local 的 CCW 弧长
  + T_local 到 T_rally 的直线段长度
  + T_rally 沿集结圆到 M_i 的 CCW 弧长
```

选择总代价最小的候选切线。

### 6.4 无解兜底

如果出现以下情况：

- 两圆重合；
- 一圆包含另一圆导致公切线无解；
- 两圆距离过近导致切线路径不可用；
- 数值退化导致候选为空；

则退回现有旧逻辑：

```text
当前位置 → 集结盘旋圆外点切入点
```

这保证开始集结命令不会因几何退化导致算法中断。

### 6.5 转场判据与交接

`RALLY_TRANSIT` 到 `RALLY_LOITER` 不是一次独立的状态跳变，而是切线转移单元把算好的集结圆切入点 `T_rally` 交给现有 `RallyJoinPos` 的过程（见 §3.3）。具体判据复用 `RallyJoinPos` 已有的 `FLYING→LOITERING` 捕获窗口（`_arc_capture_radius_m`，由 `loiter_radius_m` 反解，见 `rally_join_pos.py`）：切线转移单元飞完 `T_local → T_rally` 直线段后，通过 `RallyJoinPos` 新增的公开入口注入 `T_rally` 作为外部切入点，并把状态交给它的 `FLYING`，此时距切入点很近，通常同一拍或下一拍即满足捕获半径条件转入 `LOITERING`（对外显示切换为 `RALLY_LOITER`）。

§6.4 的几何退化兜底（当前位置直接切入集结圆外点）同样通过这条路径实现：兜底时跳过 `RallyTransit`，直接把当前位置交给 `RallyJoinPos` 走它原有的“从任意点算切入点”逻辑（`_compute_entry_point`），行为等价于关闭本方案前的旧流程。

## 7. 高度分层与收敛

### 7.1 目标

集结飞行过程中，多机可能从不同方向同时进入集结圆。为了降低近距离交汇风险，在形成松散队形之前，各机应保持高度错开；形成松散队形后，再逐步收敛到目标队形高度。

### 7.2 高度策略

完整方案采用以下阶段策略：

```text
LOCAL_LOITER / RALLY_TRANSIT / RALLY_LOITER / RALLY_EXITED / CATCHUP
  使用高度分层

LOOSE
  开始向目标队形高度收敛

COMPRESS / HOLD
  使用正常队形高度
```

**实现约束**：`CATCHUP` 阶段目前复用与 `LOOSE/COMPRESS` 相同的 `SlotGeometry` 算法（见 `leader_follower_rally/follower.py` 注释“CATCHUP 与 LOOSE/COMPRESS 用同一套算法”），而 `SlotGeometry.step()` 的高度输出是硬编码的 `leaderState.pos.h + slot.y`（`slot_geometry.py`，注释明确“高度直接保持 leader.h + slot.y，不随 scale 扩展”），没有分层覆盖入口。要让上述策略成立，`SlotGeometry` 需要新增一个可选的高度覆盖输入端口（未提供时回落到现有 `leader.h + slot.y` 行为）；`RallyTransit`/`RallyJoinPos`/`Rally` 任务在 `CATCHUP` 及更早阶段负责算出并写入分层高度，进入 `LOOSE` 后不再写入，`SlotGeometry` 自动回到默认队形高度。

### 7.3 高度分层配置

新增可选配置项：

```json
"rally_cfg": {
  "altitude_separation_m": 60.0
}
```

未配置时默认 `60.0m`。

高度分层建议围绕长机高度上下交错，而不是单调递增，避免多机数量增加时整体高度偏移过大。例如基于当前五机队形文件槽位顺序（A01、A02、A03、A04、A05，且 A03 为 `rally_leader`）：

```text
A03 leader: 基准高度
A01: 基准高度 + 60m
A02: 基准高度 - 60m
A04: 基准高度 + 120m
A05: 基准高度 - 120m
```

实际偏置顺序按队形槽位列表稳定生成：先找到 `rally_leader` 对应槽位置为 0，其余节点按槽位文件中的顺序（去掉 leader）交错分配 `+1, -1, +2, -2 ...` 个高度间隔。这样同一配置每次加载得到相同分层，也避免节点数组顺序变化影响高度分配。

**基准高度**取当前有效 `route[0]`（集结区起点 A）的高度，不取长机实时高度或长机初始配置高度：分层需要在开始集结前稳定确定，用长机实时高度会随长机爬升/下降而变化；采用避障规划结果后，覆盖航线会成为当前有效航线并重新初始化该基准。`route[0].pos.h` 与航向、切线复用同一基准点，各机在真正起飞前就能拿到，天然一致。

### 7.4 高度收敛

进入 `LOOSE` 后，目标高度切回正常队形高度：

```text
formation_altitude = leader.h + slot.y
```

完整方案可先依赖现有 `max_climb_rate_mps` / `max_descent_rate_mps` 做物理限幅，避免瞬时高度跳变。如果后续观察到高度命令突变过大，再增加高度融合因子：

```text
alt_target = mix(layer_altitude, formation_altitude, progress)
```

## 8. 界面与显示

### 8.1 新增按钮

在正式 PySide6 GUI 中新增一个控制按钮：

```text
开始集结
```

按钮状态建议：

- 未加载配置：禁用；
- 非集结配置：禁用；
- `LOCAL_LOITER`：启用“开始集结”；
- 集结中（`RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE/COMPRESS`）：“开始集结”保持可点击，点击返回“已在集结中”提示，不禁用；
- `HOLD`：禁用“开始集结”。

现有“开始/暂停/单步/重置”保留仿真运行控制语义。

### 8.2 俯视图显示

完整版本需要显示以下元素（分阶段落地时的实际交付范围见 §11，第一阶段不含切线与高度分层显示）：

- 本地盘旋圆；
- 集结盘旋圆；
- 本地圆到集结圆的切线路径；
- 切出点与切入点；
- 当前集结状态。

显示数据由 snapshot 提供，GUI 不做几何计算。沿用现有 `SimulationSnapshot.rally_geometry` 字段名，但把值类型从旧的 `RallyJoinGeometryState` 扩展为能覆盖待命与转场的 `RallyPlanGeometryState`。字段契约如下：

```text
node_id
local_center_east_m / local_center_north_m / local_radius_m       本地盘旋圆
rally_center_east_m / rally_center_north_m / rally_radius_m       集结盘旋圆
local_tangent_east_m / local_tangent_north_m                      T_local，本地圆切出点
rally_tangent_east_m / rally_tangent_north_m                      T_rally，集结圆切入点
slot_east_m / slot_north_m                                        M_i，松散目标点
fallback_used                                                     是否退回旧外点切线逻辑
```

`READY` 阶段该结构可只填本地盘旋圆、集结盘旋圆和预览切线；`start()` 之后算法按真实运行状态刷新切点与兜底标记。当前集结状态不放进几何结构，统一读取 `NodeState.rally_phase`，避免同一个状态在节点表和几何叠加里重复维护。

`fallback_used == true` 时字段仍保持非空：`local_tangent_*` 填触发兜底时的当前点，`rally_tangent_*` 填旧外点切线逻辑算出的集结圆切入点。这样 `RallyPlanGeometryState` 可以保持纯 `float` 字段，GUI 不需要处理可空切点；是否按退化样式绘制由 `fallback_used` 决定。

## 9. 模块影响

### 9.1 runner / controller

需要新增控制命令：

```text
SimulationController.start_rally()
```

`_NodeAlgorithm` 需要新增本地运行模式字段（`RallyRunModeE`，见 §3.3），并配合新增的 `FormStageE.STANDBY` 使用：

```text
rally_run_mode: STANDBY | ACTIVE
remote.stage: NONE/STANDBY/RALLY/HOLD
```

`rally_run_mode == STANDBY` 时，runner 向实体传入有效 `remote.stage == STANDBY`；`rally_run_mode == ACTIVE` 时，`remote.stage == RALLY` 内部继续由现有 Rally 任务推进 `JOINING/CATCHUP/LOOSE/COMPRESS`，不新增 `cmd.step` 值。

`_NodeAlgorithm.current_rally_phase_str()` 需要从 `rally_run_mode`、当前驱动单元、`RallyJoinPos.state` 与 `cmd.step` 派生 §3.3 的规范字符串，并写入 `NodeState.rally_phase`。该方法取代现有 `"JN·{FLY/LOIT/EXIT}"` 拼接逻辑，旧标签不再保留，避免 `_NodeAlgorithm` 内同时存在两套阶段字符串语义。`SimulationController._make_snapshot_unlocked()` 在 `run_state == READY` 时必须压制集结节点的 `rally_phase` 输出为空字符串；正式 GUI 不在 READY 阶段绘制集结辅助几何。`SimulationController._derive_control_report_unlocked()` 不能再只依赖 `current_stage()`，需要按 §3.3 的快照契约识别 `LOCAL_LOITER` 为“待命”、识别 `RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE/COMPRESS` 为“集结”。

本阶段先不做运行期几何刷新，避免在尚未实现两圆公切线转移时把静态预览误包装成真实规划结果。`SimulationController` 在初始化模块时根据配置和初始状态预计算 `SimulationSnapshot.rally_geometry`，运行中继续复用这份静态辅助几何；第二阶段实现真实 `RallyTransit` 后，再新增只读方法：

```text
current_rally_plan_geometry() -> RallyPlanGeometryState | None
```

该方法从实体内部的 `RallyJoinPos` 与后续 `RallyTransit` 读取最新本地圆、集结圆、切点和 `fallback_used`。届时 `SimulationController._make_snapshot_unlocked()` 每次生成快照时遍历集结节点调用该方法，并刷新 `SimulationSnapshot.rally_geometry`；READY 阶段仍只保留初始化时的静态几何数据，不作为可见运行阶段。

### 9.2 algorithm

位置解算槽位不新增流程对象，当前阶段统一扩展 `RallyJoinPos`：

```text
RallyJoinPos: STANDBY / FLYING / LOITERING / EXITED
RallyTransit
```

边界约定：

- `RallyJoinPos.STANDBY`：只负责当前位置本地待命盘旋，圆心由进入待命那一拍的当前位置与航向反推；它与 `LOITERING` 复用同一套“圆上投影目标 + 切向速度 + 曲率前馈 + 垂向速度”的几何口径，避免两份盘旋公式各自漂移；
- `RallyTransit`：只负责本地圆到集结圆切线转移，转移完成后把切入点交给 `RallyJoinPos`（见 §3.3、§6.5），不重复计算切入逻辑；
- `RallyJoinPos.FLYING/LOITERING/EXITED`：继续负责进入集结圆后的现有 JOINING 逻辑，后续可新增可选的外部注入切入点入口供 `RallyTransit` 交接使用；
- `SlotGeometry`：新增可选高度覆盖输入端口，供 `CATCHUP` 阶段写入分层高度（见 §7.2）。

### 9.3 通信行为

`LOCAL_LOITER` 期间僚机继续发送 `FollowerBroadcast`（不新增静默分支，保持每帧广播的现有实现方式不变）。`RallyJoinPos.STANDBY` 必须写入 `cxt.selfCmd`，使待命期间 `FollowerBroadcast.pos_err_m` 表示到本地盘旋投影目标的误差，不能沿用旧集结目标。待命期间报文中的 `rally_state` 必须显式写为 `RALLY_STATE_STANDBY`（字符串值 `STANDBY`），避免 UI/日志把待命误显示为正在飞向集结。长机在自身 `rally_run_mode == STANDBY` 时仍正常解析僚机回报，并继续通过 `RallyLeaderBroadcast` 广播 `cmd.stage = STANDBY`；任务槽位只保持 `STANDBY`，因此不会推进到“已在集结”的子流程。

实现时在 `rally_join_pos.py` 与相关测试中新增常量：

```python
RALLY_STATE_STANDBY = "STANDBY"
```

并同步更新 `FollowerBroadcastInputS.rally_state`、`FollowerStateS.rally_state` 的注释，取值范围固定为 `STANDBY / FLYING / LOITERING / EXITED`。业务代码不得直接散写裸字符串 `"STANDBY"`。

### 9.4 GUI

GUI 只新增按钮和绘制，不直接参与几何规划。控制入口通过 `ControllerSimulationAdapter` 转发到 controller。由于 `rally_geometry` 字段从旧的 `RallyJoinGeometryState` 扩展为 `RallyPlanGeometryState`，GUI 侧需要同步以下契约：

- `ControllerSimulationAdapter._convert_snapshot()`：按新字段转换本地盘旋圆、集结盘旋圆、切线路径、`T_local`、`T_rally`、`M_i` 和 `fallback_used`；
- `src/ui/gui/view_models.py` 的 `RallyGeometryView`：补齐 `local_*`、`rally_*`、`local_tangent_*`、`rally_tangent_*`、`fallback_used` 字段，旧的 `center_x/radius/entry_x` 语义不再够用；
- `src/ui/gui/top_view.py`：绘制过滤规则改为 `rally_phase in {"LOCAL_LOITER", "RALLY_TRANSIT", "RALLY_LOITER", "RALLY_EXITED", "CATCHUP"}`，不再使用旧的 `rally_phase.startswith("JN")`；`HOLD` 后不绘制集结几何；
- `LOCAL_LOITER` 显示本地盘旋圆；`RALLY_TRANSIT` 显示本地圆、集结圆和切线路径；`RALLY_LOITER/RALLY_EXITED/CATCHUP` 保留集结圆和关键点，便于观察同步等待与追赶开始前后的状态。

## 10. 测试建议

### 10.1 单元测试

几何测试：

- 当前点 + 航向 + 半径能生成正确的 CCW 本地圆；
- 两圆有可行公切线时，能选出总代价最短路径；
- 两圆重合、相交、包含时能退回旧逻辑；
- 切线两端与对应圆相切；
- CCW 弧长计算方向正确；
- `rally_geometry` 快照字段包含本地圆、集结圆、`T_local`、`T_rally`、`M_i` 和 `fallback_used`。

高度测试：

- `altitude_separation_m` 缺省为 `60.0m`；
- 多机高度分层稳定且围绕长机上下错开；
- `LOCAL_LOITER/RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP` 使用分层高度；
- `LOOSE/COMPRESS/HOLD` 回到队形高度。

状态机测试：

- `start()` 后集结节点进入 `LOCAL_LOITER`；
- `start_rally()` 从 `LOCAL_LOITER` 进入 `RALLY_TRANSIT`；
- 集结中重复 `start_rally()` 返回“已在集结中”；
- `HOLD` 后 `start_rally()` 返回 `ERR_INVALID_STATE`，不改变当前阶段；
- `LOCAL_LOITER` 时 `control_report == "待命"`，`NodeState.rally_phase == "LOCAL_LOITER"`；
- `RALLY_TRANSIT/RALLY_LOITER/RALLY_EXITED/CATCHUP/LOOSE/COMPRESS` 时 `control_report == "集结"`；
- `reset()` 后回到初始待运行状态。

### 10.2 GUI 测试

- 加载集结配置后显示“开始集结”按钮；
- 点“开始运行”后俯视图显示各机本地盘旋圆；
- 点“开始集结”后俯视图显示切线路径和集结圆；
- `HOLD` 后“开始集结”按钮禁用。

## 11. 分阶段落地建议

以下阶段是内部开发拆分；完整验收必须覆盖本方案的控制流程、切线规划、显示和高度分层。单独完成第一阶段只表示基础控制链路可运行，不代表本功能已经完成。

第一阶段：

- 新增控制命令和状态机；
- 实现本地盘旋；
- GUI 增加按钮和基础状态提示。

第二阶段：

- 实现本地圆到集结圆的切线规划；
- 输出并绘制切线、本地圆、集结圆；
- 加入几何退化兜底。

第三阶段：

- 加入高度分层；
- 调整 `LOOSE` 之后的高度收敛；
- 补充多机避撞相关回归测试。

该拆分可以降低单次改动风险，但最终行为仍按本文档的完整方案验收。
