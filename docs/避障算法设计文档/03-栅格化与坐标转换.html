03 栅格化与坐标转换

客户航线和障碍文件使用经纬度，算法内部统一使用米制 ENU。完成 LLA→ENU 后，工程再把连续 east/north 平面切成格点，供 A* 搜索。

先统一坐标层，再进入算法

层次坐标与文件职责
外部输入route_file、obstacles_file 中的 latitude / longitude / altitude面向配置、客户文件和航线输出。
配置展开以基础航线第一个航点为 origin航点、圆心、polygon 顶点使用同一原点转换。
内部规划east / north / h，单位米判障、A*、去冗余、圆弧与 GUI 画布全部使用 ENU。
航线输出内部 ENU 反算为经纬高圆弧中心与航点一起转换，保证输出文件可再次读回。
当前 JSON 航线文件要求使用经纬度航点；外部文件中的 ENU 航点会在加载期被拒绝。这里展示的 east/north 是配置展开后的内部形态，不是客户文件格式。

从世界坐标到格点

nx = ceil((max_e - min_e) / resolution_m) + 1
ny = ceil((max_n - min_n) / resolution_m) + 1

def cell_center(i, j):
    return (min_e + i * resolution_m,
            min_n + j * resolution_m)

def to_cell(point):
    i = round((point[0] - min_e) / resolution_m)
    j = round((point[1] - min_n) / resolution_m)
    return clamp(i, j)

resolution_m 是每个格点之间的米数。分辨率越小，地图越细，搜索越慢；分辨率越大，地图越粗，路径更粗糙，也更容易错过窄通道。

bounds 从哪里来

如果调用者没有传 bounds，compute_bounds() 会从本腿起点、终点和本腿候选障碍的包围盒推导搜索范围，并额外加 margin_m + resolution_m。全图远场障碍已在 planner 层预筛，不会无意义地撑大这条腿的网格。

bounds = compute_bounds(
    start,
    goal,
    obstacles,
    clearance_m,
    margin_m + resolution_m,
)
参与因素原因
起点 / 终点至少要把任务端点放进搜索范围。
候选障碍包围盒绕障空间不能只看起终点连线，但也不应被远离本腿的障碍撑大。
clearance_m障碍会膨胀，搜索范围也要覆盖膨胀后的外边界。
margin_m给绕行留额外空间。

网格规模保护

MAX_GRID_CELLS = 4_000_000

if nx * ny > MAX_GRID_CELLS:
    raise ValueError("grid too large")

这是工程保护，不是算法参数。用户把分辨率调得过小或者范围调得过大时，网格数量会爆炸，A* 的内存和耗时都会失控，所以这里直接把配置错误暴露出来。

粗略感受：同一范围内，分辨率减半，横向格点约翻倍、纵向格点约翻倍，总格点约变成 4 倍。
