# 航线文件设计

## 目标

`route_file` 是面向客户场景的统一航线输入边界。内部算法、仿真控制和避障后端只消费展开后的 `config["route"]`，不直接感知外部航线文件格式。控制器初始使用该配置航线；用户采用避障规划结果后，覆盖航线成为当前有效航线。集结场景使用当前有效航线首点作为集结中心、首段作为集结方向，并在集结完成后继续使用后续航段。

## 设计原则

- 主配置只声明 `route_file` 相对路径。
- 相对路径统一按主配置文件所在目录解析，不依赖当前工作目录。
- 航线文件策略只负责“读/写某种文件格式”，不做仿真语义解释。
- `LineFileManager` 负责路径解析和策略调度。
- `LineFileStrategyFactory` 负责选择具体策略；后续客户格式只新增策略并注册到工厂。

## 当前策略

`JsonLineFileStrategy` 支持 `.json` 文件，文件根对象就是完整的 `route` 结构。当前对外基础航线必须使用经纬高：

```json
{
  "speed_mps": 20.0,
  "waypoints": [
    {"latitude_deg": 39.0, "longitude_deg": 116.0, "altitude_m": 1000.0, "R": 0.0}
  ]
}
```

加载时以上述 `waypoints[0]` 作为 ENU origin。该 origin 不写入主配置文件，仅存在于加载后的配置副本中，供障碍经纬度转换和避障航线输出复用。高度不参与坐标转换，`altitude_m` 原样进入内部高度字段。

对外 `route_file` 不再接受旧 ENU 字段 `x_m/y_m/altitude_m`：加载期会明确报错，避免客户文件和内部坐标边界继续混用。经纬高航线展开后，内存配置副本才会增加 `x_m/y_m/altitude_m` 供算法、控制器和 GUI 使用。`route_inputs_to_config()` 在缺少地理原点时仍可返回 ENU 对象供内部测试或兼容调用，但这种对象不属于当前可再次加载的对外航线合同；正式 GUI 从已加载航线取得 origin，输出经纬高。

避障航线输出可能包含已烘焙圆弧航段，此时航点会额外带 `turn_sign` 与经纬度 `center`：

```json
{
  "latitude_deg": 39.001,
  "longitude_deg": 116.002,
  "altitude_m": 1000.0,
  "R": 0.0,
  "turn_sign": 1.0,
  "center": {"latitude_deg": 39.002, "longitude_deg": 116.002, "altitude_m": 1000.0}
}
```

控制器读取 `route.waypoints[]` 时兼容这些字段：`turn_sign` 为 `0` 或缺省时仍按普通折线和 `R` 处理，非 `0` 时按已知圆弧航段处理。

`DiamondXmlLineFileStrategy` 支持钻石项目 `.xml` / `.XML` 航线文件。输入文件结构按客户 XML 校验：根节点为 `Root`，含 `Item` 元信息和连续的 `Item1..ItemN` 航点节点；不校验文件名，也不校验输入 `SkywayNo`、`SkypointNo` 的具体取值。钻石 XML 不含速度字段，加载后由策略补 `speed_mps = 45.0`。

钻石 XML 输出固定为避障航线规范：`SkywayNo=25`、`SkypointNo=1`、`ByLineName=芜湖自动避障航线`、`StLine_Type=0`，`IdAllNum` 等于输出航点数量，`CreatTimer` 和文件名使用同一次生成时间。实际文件名由策略规范化为 `航线25 芜湖自动避障航线 YYYY年M月D日H时M分S秒.XML`，GUI 保存对话框也通过策略获取同格式默认文件名。钻石格式只承载经纬高航点，若输出 route 中含 `turn_sign` / `center` 这类真实圆弧航段字段，策略直接报错，避免把客户格式不支持的圆弧静默丢失。

## 扩展方式

新增客户格式时：

1. 新增一个实现 `LineFileStrategy` 的策略类。
2. 在工厂注册该策略。
3. 策略的 `load()` 返回兼容 `config["route"]` 的对象。
4. 策略的 `save()` 接收同一结构并输出客户需要的文件格式。

这样控制器、GUI 和算法层不需要跟随客户文件格式变化。
