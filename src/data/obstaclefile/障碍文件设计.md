# 障碍文件设计

## 目标

`obstacles_file` 是面向客户场景的障碍输入边界。内部避障后端仍只消费展开后的 `config["avoidance"]["obstacles"]`，不直接感知外部障碍文件格式。

## 设计原则

- 主配置只在 `avoidance.obstacles_file` 声明相对路径。
- 相对路径统一按主配置文件所在目录解析，不依赖当前工作目录。
- 障碍文件策略只负责“读/写某种文件格式”，不做避障语义解释。
- `ObstacleFileManager` 负责路径解析和策略调度。
- `ObstacleFileStrategyFactory` 负责选择具体策略；后续客户格式只新增策略并注册到工厂。

## 当前策略

`JsonObstacleFileStrategy` 支持 `.json` 文件，文件根可以是原 `base.json.avoidance.obstacles` 的数组。对外障碍位置使用经纬度；圆形半径仍使用米：

```json
[
  {
    "id": "C1",
    "type": "circle",
    "enabled": true,
    "center": {"latitude_deg": 39.0, "longitude_deg": 116.01},
    "radius_m": 200.0
  }
]
```

矩形障碍使用四个点的经纬度，允许旋转矩形：

```json
{
  "id": "R1",
  "type": "rect",
  "enabled": true,
  "points": [
    {"latitude_deg": 39.0, "longitude_deg": 116.0},
    {"latitude_deg": 39.0, "longitude_deg": 116.01},
    {"latitude_deg": 39.01, "longitude_deg": 116.01},
    {"latitude_deg": 39.01, "longitude_deg": 116.0}
  ]
}
```

障碍文件策略只读出原始数组；经纬度到 ENU 的转换由 `config_loader` 在加载基础航线后注入 origin 完成，障碍文件自身不查航线信息。

也允许对象根，但必须显式包含 `obstacles` 键；缺键视为配置错误，而不是静默退化为空障碍。

## 扩展方式

新增客户格式时：

1. 新增一个实现 `ObstacleFileStrategy` 的策略类。
2. 在工厂注册该策略。
3. 策略的 `load()` 返回兼容 `avoidance.obstacles` 的数组。
4. 策略的 `save()` 接收同一数组并输出客户需要的文件格式。

这样控制器、GUI 和避障算法层不需要跟随客户文件格式变化。
